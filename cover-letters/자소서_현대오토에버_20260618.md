## [1. 지원동기]

지난 2년 5개월간 ㈜마이아이비에서 기부·모금·봉사 플랫폼의 프론트엔드를 담당하며, 신뢰가 흔들리는 순간 사용자가 바로 떠난다는 사실을 매일 코드로 마주했습니다. 후원금이 한 번 잘못 결제되면 사용자는 다시 돌아오지 않고, 단체가 한 번 부정확한 정보로 노출되면 플랫폼 전체에 대한 신뢰가 영향을 받습니다. 그래서 단체 상세 페이지 PRD 작성부터 디자인 시스템 토큰화, Sentry 누적 에러 전수 조사까지 화면 한 장의 가설을 정의하는 일부터 그 화면이 안정적으로 돌아가게 만드는 일까지를 한 사람이 이어서 책임지는 방식으로 일해 왔습니다.

현대오토에버를 지원한 이유는 "현대차그룹의 SW 역량과 그룹 내외 로봇을 결합해 로봇 인텔리젼스 서비스를 가능케 한다"는 RX추진실의 방향이 제가 myib에서 익힌 일하는 방식과 맞닿는 부분이 많기 때문입니다. 이기종로봇관제시스템과 통합관제 시각화 도구(대시보드·실시간 모니터링)·대규모 Fleet 관리처럼 현장의 안전과 생산성이 화면 한 장의 정합성에 직결되는 영역에서는 제품 표면의 안정성과 데이터 정합성이 그대로 고객 만족의 근거가 된다고 생각합니다. 2024년 신설된 RX추진실이 현대차그룹의 미래 먹거리인 로봇 분야를 본격 추진하는 시점이라는 점에서, 새 시스템을 0→1로 셋업해 본 경험과 임팩트 도메인에서 쌓은 신뢰 중심의 프론트엔드 아키텍처 설계·운영 경험을 바탕으로, 로봇 통합 관제 SW 플랫폼 영역에서 현대오토에버의 제품 가치를 함께 고도화하고자 지원했습니다.

## [2. 직무 경험 ① - 디자인 시스템을 코드에서 끌어올린 1년]

myib 입사 시 코드베이스는 hex 컬러가 수백 곳에 흩어져 있고, native alert() 호출이 58개, 어드민 테이블은 6종이 비표준으로 운영되고 있었습니다. 단순히 컴포넌트를 새로 만드는 것이 아니라, 기존 코드에 이미 형성된 패턴을 정확히 읽어내는 것에서 시작해야 한다고 판단했습니다. 디자이너에게 "top-down으로 새 시스템을 받아오자"고 하기 전에, 코드 안의 실제 사용처를 카테고리별로 추출하는 bottom-up 접근을 택했습니다.

이 원칙으로 약 1년에 걸쳐 banner·button·icon·selector·table·toast 등 20여 개 카테고리를 토큰화하고, Storybook을 구축해 GitHub Pages에 자동 배포되도록 CI를 세팅했습니다. 디자이너·개발자가 매번 새로 합의하지 않아도 되도록 Figma↔코드 매핑을 단일 진실 문서(DESIGN.md)로 도입했습니다. 어드민 영역에서는 DataTable·StatusBadge를 신규 구축한 뒤 두 번째 내부 어드민 시스템(admin2)의 26개 테이블 중 22개를 표준 컴포넌트로 교체했고, 흩어져 있던 native alert 58건은 일관된 showToast 패턴으로 일괄 전환하면서 사용되지 않던 useMessage 및 MessageContext 레이어까지 함께 제거했습니다.

이와 병행해 레거시 admin 디렉토리 190개 파일을 외부 의존성과 함께 단일 commit으로 이관·정리했고, 하루 동안 미사용 컴포넌트·유틸·public 정적 파일 약 320여 개를 일괄 삭제했습니다. 아이콘 영역에서는 카테고리별 100여 개를 약 60개로 통합(40% 절감)하면서 PNG 16종을 SVG 컴포넌트로 전환했고, 버튼은 FillButton 단일 표준으로 통일하며 90개 파일을 한 번에 교체해 코드베이스의 부피와 빌드 비용을 함께 줄였습니다.

이 작업은 화려한 신규 기능은 아니었지만, 결과적으로 B2C 앱·B2B 파트너·어드민 3개 프로덕트가 같은 토큰과 컴포넌트로 수렴하면서 신규 화면 하나를 만드는 데 드는 의사결정 비용이 줄었습니다.

## [3. 직무 경험 ② - 기획서를 직접 쓰고 그 위에서 코드를 짠다]

프론트엔드 개발자로 일해왔지만, 화면 한 장을 만들 때마다 사용자가 어디서 들어와 무엇을 결정하고 어디로 빠져나가는지를 먼저 그려보는 사람입니다. 기능 명세를 받아 구현하기보다, 유저의 의사결정 동선·고객사(단체·기업)의 노출 요구·웹에서의 표현 방식 세 축을 한 문서에서 함께 설계한 뒤 자기 손으로 구현하는 사이클로 일합니다. 의견이 갈리는 상황에서도 화면 한 장이 놓일 시스템 전체에서 무엇이 맞춰져야 하는지를 함께 보고 하나씩 합의를 만들어가는 방식이, 화면이 제 기능을 하도록 만든다고 믿습니다.

기획서의 입력은 고객지원 채널에 누적된 사용자 문의, 진입 경로별 동선 분석, 백엔드·운영·마케팅 담당자와의 직접 협의 세 채널에서 옵니다. 대표 사례인 단체 상세 페이지 리디자인 PRD(v2.0)에서는 "어떤 단체인지 신뢰가 안 간다", "기부 버튼이 어디 있는지 모르겠다" 같은 CS 응대 기록을 출발점으로 '신뢰 신호 분산 / CTA 휘발성 / 임팩트 추상' 세 가지 문제를 정의했고, 화면 단위가 아니라 진입, 검토, 결정, 결제로 이어지는 퍼널(Funnel) 전체에서 North Star와 7가지 하위 KPI(Sub-KPI)를 잡았습니다. 페르소나를 세 그룹으로 세분화하고, 4단계 결제 흐름을 Mermaid 플로우 다이어그램과 7개 화면 인터랙티브 프로토타입으로 옮겨 동료들이 클릭만으로 가설을 검토할 수 있도록 했습니다. 마이페이지 기획서에서는 9개 메뉴 × 5개 상태 케이스(약 45 조합)를, 모금 상세 리디자인에서는 모금 상태·매칭·신뢰도·콘텐츠를 조합한 100여 가지 케이스를 같은 방식으로 정리했습니다.

동일한 책임 범위를 바탕으로 B2B 파트너 프론트엔드 시스템을 Next.js 기반으로 제로 베이스에서 단독으로 셋업하며 크로스 기능 조직 간 도메인 인터페이스 설계 합의를 주도했습니다. Apollo Client v4 마이그레이션 시 MockedProvider 사양 변경과 타입 캐스팅 이슈를 해결하고, 서버 장애로 직결될 수 있는 deprecated GraphQL 쿼리 10건을 신규 스키마로 안전하게 이관했으며, Sentry 누적 에러 23건의 전수 조사와 예외 처리 보완까지 완수해 화면 위의 가설부터 하부 데이터·관측성 레이어의 정합성까지 전 과정을 책임지는 엔지니어링 역량을 증명했습니다. 결제 view에서는 주민등록번호·카드 비밀번호 등 민감정보 입력 검증(MIME/확장자 포함)과 SSL 표기 정합성을 정리하면서, 백엔드 세션 만료를 안정적으로 추적하는 폴링 구조를 직접 설계해 미인증 API 호출 가드와 에러 기반 리다이렉트까지 한 사이클로 묶었습니다. 검증 사이클 측면에서는 파트너웹 로그인·회원가입·약관·정본 Modal primitive를 mobile-first 구조로 재작성하면서 Playwright·chrome-devtools MCP를 검증 워크플로우에 도입해 360px·1280px 양쪽에서 dvh·safe-area-inset-bottom·sticky CTA bar 동작을 동시에 확인하고 콘솔 에러 0을 확보했고, 검색 노출과 성능 측면에서는 Next.js App Router의 generateMetadata 동적 API를 활용해 단체·모금·봉사·캠페인 등 도메인별 메타데이터와 Open Graph 정보를 일관된 SSR 구조로 정리하면서 같은 맥락에서 Core Web Vitals 관점의 빌드 비용·번들 최적화(레거시 320여 개 정리, 아이콘 100→60 통합, FillButton 단일화로 90개 파일 결합 구조 일괄 교체) 역시 한 사이클로 묶어 진행했습니다.

## [4. 성격과 일하는 방식 - 정밀함·완결성과 협업, AI 레버리지]

성격의 가장 큰 특징은 모호한 현상에 머무르지 않고 근본적인 원인을 추적하여 체계화하는 정밀함과 완결성입니다. 컴포넌트 배치 오차를 스타일 래퍼로 임시 봉합하는 대신 SVG 자체의 viewBox를 통일하여 해결하는 것처럼, 한 번 정립한 구조적 원칙을 코드베이스 전체에 일관되게 적용하려 합니다. 예외 처리 미비로 발생할 수 있는 잠재 장애를 막기 위해 센트리 에러 23건을 전수 조사하고 하루 만에 해결한 것 또한 이러한 완결성 중심의 성격에서 비롯되었습니다.

이러한 정밀한 성격은 협업을 할 때 말로 앞서기보다 작동하는 시스템과 정제된 문장으로 소통하는 방식으로 발현됩니다. 1인 프론트엔드 개발자로서 다양한 직군과 협업하며, 서로의 언어가 달라 발생하는 싱크 오류를 줄이는 데 집중해 왔습니다. 디자이너와의 커뮤니케이션 비용을 줄이기 위해 피그마와 코드를 일원화한 단일 진실 문서를 도입하고, 백엔드 개발자와의 데이터 규격을 맞추기 위해 PRD에 엔지니어링 주석을 달아 사전에 합의하는 방식이 저의 대표적인 협업 스타일입니다. 주관적인 의견 대립이 생기더라도 데이터 지표와 퍼널 다이어그램을 제시하며 전사적인 합의를 주도적으로 이끌어냅니다. RX추진실 동료의 한 마디 중 "어려운 문제를 해결하기 위해 다양한 시각에서 고민하고, 자유로운 분위기에서 활발하게 의견을 나눠요" 라는 문구는 이러한 협업 방식과 결을 같이한다고 느꼈습니다.

반면 시스템 전체의 구조와 큰 그림을 먼저 조망하려 하기에 초기 아키텍처 설계 단계에서 다소 신중해진다는 점이 있습니다. 이러한 신중함을 보완하고 비즈니스의 속도를 맞추기 위해 저는 두 가지 레버리지를 활용합니다.

첫째는 고속 실행이 필요한 타이밍에 의도적으로 업무 모드를 전환하는 메타인지이며, 둘째는 Claude와 Gemini 같은 AI 도구를 가상 아키텍처 파트너로 워크플로우에 내재화하는 능력입니다. 단, 현장의 안전과 생산성이 실시간 데이터 정합성에 직결되는 도메인 특성을 고려하여 AI가 제안한 구조나 데이터 치환 결과물을 무비판적으로 수용하지 않고, 저의 정밀한 성향을 바탕으로 철저한 교차 검증 단계를 거쳐 기술적 할루시네이션이나 결함 리스크를 통제합니다. 이러한 기술 제어를 통해 신중함이 만드는 속도의 제약을 생산성 저해 없이 극복하고 있습니다.

## [5. 입사 후 포부]

현대오토에버에서 함께 일할 기회를 주신다면, 제 주장을 앞세우기보다 가장 먼저 팀의 소통 방식과 기존 코드베이스의 문화를 온전히 흡수하는 데 집중하겠습니다. 새로운 조직에 합류할 때 가장 중요한 것은 동료들에게 누를 끼치지 않고, 기존 시스템의 안정적인 운영에 빠르게 보탬이 되는 것이라 생각합니다. 이기종로봇관제시스템, 통합관제 시각화 도구, 대규모 Fleet 관리 등 제조·물류 현장의 생산성과 서비스 부문의 안전·편의성을 책임지는 핵심 플랫폼에 성실히 적응하여, 기존 팀원분들이 쌓아 올린 견고한 토대 위에 묵묵히 제 몫의 개발 리소스를 더하겠습니다. 로봇 통합 관제 도메인의 특성상 이기종 로봇·외부 시스템·실시간 데이터가 한 화면 위에서 동시에 흐르는 만큼, 자만하지 않고 기초부터 철저히 배우며 팀의 일원으로서 완벽히 녹아들겠습니다.

이후 비즈니스 방향성에 맞추어 제가 가진 경험들이 팀원분들의 업무 부담을 실질적으로 덜어주는 무기가 되기를 바랍니다. 마이아이비에서 제로 베이스부터 단독으로 시스템을 구축하고 요구사항 문서(PRD)를 주도적으로 다듬어본 경험은, 2024년 신설된 RX추진실이 현대차그룹의 로봇 사업을 본격 추진하고 이기종 로봇관제 플랫폼을 0→1로 정착시키는 단계에서 그대로 옮겨갈 수 있는 자산이 될 것입니다. 상태 관리 측면에서는 server state를 다루는 Apollo Client(v3→v4 마이그레이션 전담)와 client state를 다루는 Zustand를 두 레포에서 14개 store(결제·시스템·사용자·기부·봉사·주민번호 입력·토큰 등)로 분리 운영하며 책임 경계를 명확히 두는 패턴을 정착시켰고, B2C 앱·B2B 파트너·어드민 3개 사용자 그룹의 결제·신원 인증·민감정보 흐름을 한 명이 책임지며 데이터 정합성을 운영해 본 경험은 다양한 로봇·외부 시스템·실시간 데이터가 한 화면 위에서 동시에 흐르는 통합 관제 플랫폼 운영에 그대로 이어갈 수 있다고 생각합니다. 또한 평소 연구해 온 AI 협업 워크플로우와 Playwright·chrome-devtools MCP 기반 검증 사이클 역시 혼자만의 생산성에 머무르지 않고, 팀의 전체적인 개발 손길을 가볍게 만들고 동료들과 다 함께 성장하는 데 조용히 기여하고 싶습니다.

"로봇 인텔리젼스 서비스로 고객의 효율성과 경쟁력을 최대화한다"는 RX추진실의 방향, 그리고 "새로운 시각으로 세상을 바라보고 이를 다양한 기술과 접목시켜 사업화까지 완료한다"는 팀 리더의 미션은 결코 혼자서 만들 수 없다고 믿습니다. 팀원들과 긴밀히 협업하고 끊임없이 배우며, 현대오토에버가 지향하는 가치를 사용자가 마주하는 화면에서 완벽한 신뢰성으로 구현해 나가는 가장 성실하고 든든한 동료가 되겠습니다.
