SELF-INTRODUCTION · 자기소개서
01
지원동기

지난 2년 5개월간 ㈜마이아이비에서 기부·모금·봉사 플랫폼의 프론트엔드를 담당하며, 신뢰가 흔들리는 순간 사용자가 바로 떠난다는 사실을 매일 코드로 마주했습니다. 후원금이 한 번 잘못 결제되면 사용자는 다시 돌아오지 않고, 단체가 한 번 부정확한 정보로 노출되면 플랫폼 전체에 대한 신뢰가 영향을 받습니다. 그래서 단체 상세 페이지 PRD 작성부터 디자인 시스템 토큰화, Sentry 누적 에러 전수 조사까지 화면 한 장의 가설을 정의하는 일부터 그 화면이 안정적으로 돌아가게 만드는 일까지를 한 사람이 이어서 책임지는 방식으로 일해 왔습니다.

컬리를 지원한 이유는 간편 결제를 넘어 혁신적인 금융 경험을 만들어 고객의 일상을 편리하게 바꾼다는 핀테크 조직의 미션이, 제가 매일 다뤄 온 화면의 성격과 그대로 겹치기 때문입니다. 기부 플랫폼의 결제 화면은 「쇼핑」이라는 말이 붙지 않았을 뿐, 사용자가 자기 돈을 맡기는 순간이라는 점에서 동일합니다. 정기결제가 한 번 실패하고 그 사실이 화면에 드러나지 않으면 사용자는 서비스를 신뢰하지 않고, 결제 완료 화면에서 금액과 다음 결제일이 모호하면 그 자리에서 이탈합니다. 저는 그 화면을 직접 기획하고 직접 구현하며 결제 표면의 정확성이 곧 제품 신뢰라는 것을 배웠습니다.

컬리페이의 웹 프론트엔드, AML 시스템, 결제 SDK·서드파티 연동 페이지는 각각 고객 화면·내부 운영 화면·외부 연동이라는 세 가지 성격을 동시에 요구하는 영역이라 생각합니다. myib에서 B2C 앱·B2B 파트너·내부 어드민 세 프로덕트를 한 사람이 함께 운영하며 결제·정산·포인트 도메인을 모두 다뤄 본 경험을 바탕으로, 컬리의 핀테크 제품 가치를 함께 고도화하고자 지원했습니다.

02
직무 경험 ① — 결제·정산·포인트, 돈이 오가는 화면을 책임져 왔습니다

결제 흐름에서 가장 먼저 손댄 것은 결제 완료 화면이었습니다. 정기/일시 결제·매칭·영수증 분기가 뒤엉켜 409줄까지 자란 화면을, 상태 조합부터 다시 정의해 134줄의 단순한 구조로 재설계했습니다. 이때 분기를 없애 단순하게 만든 것이 아니라 오히려 정기/일시 결제별 헤딩 카피 분기와 「매월 N일 결제 예정」 같은 사용자에게 꼭 필요한 정보는 새로 추가했습니다. 회사 모금과 파트너 모금이 각각 갖고 있던 결제 완료 화면도 공용 뷰(CheckoutSuccessView)로 일원화해, 결제 완료 경험이 진입 경로와 무관하게 동일하도록 정리했습니다.

결제는 성공보다 실패를 어떻게 드러내는가가 신뢰를 만든다고 생각합니다. 정기기부 결제 실패를 운영자와 사용자 양쪽에서 가시화하는 작업을 진행하며, 어드민 목록에는 실패 건수 pill을, 상세 모달에는 실패 배지·사유·재시도 예정을 노출하고 중복 집계되던 상태 카운트를 배타적으로 정리했으며, 사용자 마이페이지에도 정기결제 실패 상태를 드러냈습니다. 결제 view 안에서는 주민등록번호·카드 비밀번호 같은 민감정보 수집 양식의 입력 검증(MIME·확장자 검증 포함)과 SSL 표기 정합성을 정리하고, 결제 결과를 기다리는 구간의 안전한 세션 폴링 구조를 확립했습니다.

결제 이후 단계인 정산과 포인트도 직접 구축했습니다. B2B 파트너 정산(Billing) 도메인을 1차 퍼블리싱부터 단독으로 구축하며 정산서 조회·상세 내역 화면, 정책 패널, 계좌 등록/변경 모달을 스펙에 맞춰 구현했고, 돈과 직결되는 화면인 만큼 dev는 전체 오픈하되 production은 허용 파트너 화이트리스트에만 정산 메뉴와 /billing 라우트를 노출하는 환경별 게이팅을 URL 가드까지 포함해 구현했으며, 법인 표의 민감 컬럼 노출도 함께 정리했습니다. 기부 확인서 PDF(미리보기·로고 워터마크·신청/완료 자동 구분)와 엑셀 출력은 exceljs 공용 빌더로 통합했고, 마이페이지에서는 기부·포인트·매칭 내역 3종을 실데이터로 통합했습니다(포인트 내역 563줄 재작성). 임팩트는 결제·정산·포인트가 각각 다른 규칙으로 흩어져 있던 상태에서 하나의 일관된 표면으로 정리된 것이고, 러닝 포인트는 금액이 걸린 화면에서는 「보여주는 것」만큼 언제·누구에게 보여주지 않을지를 설계하는 일이 중요하다는 것입니다.

03
직무 경험 ② — 반복되는 업무·코드·장애 요소를 구조적으로 해결하다

반복은 언젠가 장애가 된다고 생각합니다. myib 입사 시 코드베이스에는 hex 컬러가 수백 곳에 흩어져 있고, native alert() 호출이 58개, 어드민 테이블은 6종이 비표준으로 반복되고 있었습니다. 하나씩 고치는 대신 반복 패턴을 읽어내 표준 하나로 수렴시키는 방식을 택해, alert 58건은 showToast 단일 패턴으로 일괄 전환하며 그 아래 useMessage·MessageContext 레이어까지 제거했고, 테이블 6종은 DataTable·StatusBadge 두 컴포넌트로 추상화해 어드민 테이블 26개 중 22개를 교체했습니다. 이렇게 만든 표준이 다시 흐트러지지 않도록 20여 카테고리를 토큰화하고 Storybook 자동 배포 CI와 Figma↔코드 단일 진실 문서(DESIGN.md)를 세웠으며, 두 레포의 토큰 집합을 「시각 변화 0」 전제로 완전히 일치시킨 뒤 baseline(ratchet) 방식 lint:tokens와 CI 게이트를 양쪽에 이식했습니다. 새 위반은 통과할 수 없고 기존 위반만 줄어드는 구조라, 규칙이 문서가 아니라 파이프라인으로 지켜집니다.

장애 요소도 같은 방식으로 다뤘습니다. SSR·서버 페치에서 산발적으로 나던 401 오류는 재시도로 덮지 않고 NextAuth jwt 콜백의 만료 토큰 갱신 누락이라는 근본 원인을 찾아 수정했고, refresh token rotation에서 새 토큰 전파 누락으로 발생하던 조기 로그아웃도 원인부터 해결한 뒤 rememberMe 세션 쿠키 Max-Age 정규화 모듈을 TDD로 구현했으며, proxy 레이어를 순수 route guard로 축소해 인증 책임을 단일화했습니다. 파트너 시스템에서는 Next.js 미들웨어로 세션 토큰 갱신·미인증 API 호출 가드·에러 기반 리다이렉트 분기를 직접 제어했습니다. 관측성 측면에서는 4개 레포의 Sentry 설정을 단일 표준으로 통일하고 누적 에러 23건을 전수 조사해 잠재 크래시를 사전 제거했으며, 인증 헤더의 PII 스크럽 규칙을 양 레포에 통일해 민감정보가 모니터링 경로로 새지 않도록 막았습니다. 의존성 취약점은 63건 → 10건(critical/high 전부 제거, swiper prototype pollution 등 패치), eslint error는 97건 → 0건으로 정리했습니다.

AML처럼 규정과 데이터 정합성이 핵심인 내부 시스템 경험도 인접합니다. 두 번째 어드민(admin2)에서 모금 캠페인 검수와 파트너별 게시 게이팅 화면을 신설하며 「무엇을 통과시키고 무엇을 막을지」를 화면에서 결정하는 구조를 다뤘고, GraphQL 응답을 미러링하던 Zustand store를 걷어내고 useQuery 단일 소스로 재편해 server state와 client state의 경계를 정리했습니다. Apollo Client v3→v4 마이그레이션을 전담하며 서버 사양에서 제거되면 런타임 장애로 직결될 deprecated 쿼리·필드 10건을 백엔드와 협의해 안전하게 이관한 것도, 장애가 나기 전에 없애는 일이 결국 가장 싸다는 판단에서였습니다.

04
일하는 방식 — Core Web Vitals는 실측으로, AI는 검증을 붙여서

성능은 체감이 아니라 숫자에서 출발해야 한다고 생각합니다. 아무도 요청하지 않았지만 프로덕션 홈을 직접 실측해 네트워크 페이로드 18.5MB, LCP 리소스 로드 지연 5,050ms라는 기준선을 만들고 P0~P3 백로그를 스스로 세웠습니다. 그 과정에서 앱 전체 SSR 마크업이 비어 있던 근본 원인이 세션 로딩 게이트에 있다는 것을 추적해 서버 렌더를 root HTML 0자 → 95KB로 복구했고, LCP 배너를 서버 페치로 전환, 폰트 2MB를 dynamic subset으로 축소, 렌더 차단 체인 680ms를 제거했습니다. 파트너웹에서는 xlsx를 ESM 엔트리 지연 로딩으로 전환해 정산 화면 초기 번들을 줄였는데, bare 지연 로딩 시 SSR에서 require is not defined 500 오류가 나는 원인까지 진단해 해결했습니다. 브라우저 렌더링 파이프라인을 이해한다는 말은, 저에게는 이렇게 병목을 특정하고 근본 원인까지 내려가 본 경험을 뜻합니다.

모바일 환경도 실측으로 다룹니다. login·signup·legal 화면과 정본 Modal을 mobile-first로 재작성하고 chrome-devtools 기반으로 360/1280 양쪽을 검증해 콘솔 에러 0을 확보했으며, Modal 프리미티브 하나를 바꿔 전 서비스 모달이 자동으로 모바일에 대응되도록 만들었습니다. 색 대비 AA·랜드마크·헤딩 위계 보정도 같은 사이클에서 함께 진행했습니다.

문제 정의의 출발점은 사용자의 목소리입니다. 단체 상세 리디자인은 「어떤 단체인지 신뢰가 안 간다」, 「기부 버튼이 어디 있는지 모르겠다」는 CS 응대 기록에서 출발해 문제 세 가지를 정의하고, 진입→검토→결정→결제 퍼널 전체에 North Star와 7개 하위 KPI를 잡은 뒤 그 기획안대로 직접 구현했습니다. 프론트엔드 개발자지만 화면 한 장의 가설을 스스로 쓰고 그 위에서 코드를 짜는 방식으로 일해 왔고, 결제 전환 흐름처럼 비즈니스 숫자가 걸린 영역에서 이 방식이 가장 잘 작동했습니다.

AI는 이미 업무의 기본 레이어입니다. 다만 쓰는 것보다 검증을 붙이는 것에 무게를 둡니다. 워크스페이스 루트에 AI 에이전트가 읽는 규칙 문서(CLAUDE.md)를 정본으로 두고 「단순함 우선 / 임시방편 금지, 근본 원인까지 / 최소한의 영향」 같은 원칙과 컨벤션·아키텍처 규칙을 명문화했으며, 규칙 위반은 앞서 말씀드린 lint:tokens CI 게이트에서 걸리도록 자동화했습니다. 화면 작업 루틴은 DESIGN-CHECK.md 한 벌로, 작업 인계와 회고는 NEXT-SESSION.md·refactor-log.md·docs/lessons/로 남깁니다. 덕분에 대량의 레거시 타입 치환처럼 위험한 작업을 AI로 가속하면서도 교차 검증 단계를 통해 할루시네이션과 잠재 결함 리스크를 통제할 수 있었습니다. 제품 안에서도 모금 만들기 화면에 LLM 자동채우기를 연동하고, 파트너웹 composer에서 PDF 첨부→AI 요약 흐름을 일원화했습니다. 결제·금융 도메인일수록 AI 산출물을 그대로 신뢰하지 않는 이 습관이 더 중요하다고 생각합니다.

05
입사 후 포부

컬리에서 함께 일할 기회를 주신다면, 가장 먼저 팀의 소통 방식과 컬리페이·AML 코드베이스에 형성된 맥락을 정확히 읽는 데 집중하겠습니다. 합류 초기의 가장 빠른 기여는 기존 시스템의 안정적인 운영에 보탬이 되는 것이라 생각하기 때문입니다. 결제 도메인의 정산·포인트·선불 흐름을 빠르게 흡수하고, 제가 검증해 온 운영 역량 — 관측성 정비, 예외 처리 보강, 성능 기준선 수립, 민감정보 처리 점검 — 부터 제 몫의 기여를 시작하겠습니다. 이후에는 디자인 시스템과 공통 모듈 영역에서, 여러 프로덕트를 하나의 언어로 수렴시켜 본 경험으로 팀원분들의 반복 작업을 구조적으로 덜어내겠습니다.

솔직하게 말씀드릴 부분도 있습니다. 경력은 2년 5개월로 3년 요건에 조금 못 미치지만, 그 기간 동안 4개 프로덕트의 프론트엔드를 단독으로 운영하며 0→1 구축부터 결제·정산·인증·성능·보안까지 한 사람이 끝까지 책임지는 경험을 밀도 있게 쌓았고, 「이에 준하는 실력」으로 평가받을 자신이 있습니다. 기술적으로는 Nx·Turborepo 모노레포와 NestJS BFF, Kubernetes·MSA 운영이 직접 경험이 없는 영역입니다. 다만 두 레포의 토큰·도구 체인·문서를 동기화하고 CI 게이트를 양쪽에 이식하며 멀티 레포 거버넌스를 직접 설계해 본 만큼 모노레포가 풀려는 문제는 손으로 겪어 봤고, Next.js 서버 환경에서 다뤄 온 Node 기반 API·렌더 계층과 TypeScript 데코레이터 패턴이 NestJS의 진입 지점이 되리라 생각합니다. Docker로 개발 환경을 다뤄 온 경험 위에 컨테이너·배포 파이프라인을 학습해 접점을 넓히겠습니다. 앱 내 웹뷰 역시 전용 제품 경험은 없으나, mobile-first 재작성과 360 폭 콘솔 에러 0 검증을 직접 해 온 감각이 출발점이 될 것입니다.

기술을 통해 비즈니스의 가치를 극대화하고 고객의 일상을 편리하게 바꾼다는 미션은 결코 혼자서 만들 수 없다고 믿습니다. 팀원들과 긴밀히 협업하고 끊임없이 배우며, 컬리가 지향하는 가치를 고객이 마주하는 결제 화면에서 흔들리지 않는 신뢰성으로 구현해 나가는 가장 든든한 동료가 되겠습니다.
