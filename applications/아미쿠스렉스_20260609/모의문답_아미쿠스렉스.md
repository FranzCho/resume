## A. 오프닝 — 어느 면접에서나 첫 질문

### Q. 간단히 자기소개를 해주세요. (1분 버전)
> "프론트엔드 엔지니어 조성민입니다. 마이아이비라는 기부·모금 플랫폼에서 2년 4개월간 1인 프론트엔드로 B2C 앱, B2B 파트너, 내부 어드민, 트랜잭션 메일 4개 프로덕트를 단독 운영했습니다. 4개 레포에 2,200커밋 정도 쌓았고요. 제가 한 일을 세 줄로 요약하면 — 첫째, hex와 비표준 컴포넌트가 흩어져 있던 코드베이스를 1년에 걸쳐 디자인 시스템으로 토큰화했고, 둘째, B2B 파트너 프론트엔드를 Next.js로 제로베이스 셋업했으며, 셋째, 모금 만들기 화면의 LLM 자동채우기처럼 AI 출력을 사용자가 검토·수정하는 인터페이스를 다뤘습니다. 로폼이 찾는 'Next.js B2B + Radix·Tailwind 디자인 시스템 + AI 기능' 세 축이 제가 해온 일과 거의 그대로 겹쳐서 지원했습니다."

## B. 기술 면접 (구두 Q&A)

### Q1. Next.js App Router와 Pages Router, 언제 무엇을 쓰나요?
> "신규로 시작하거나 SSR 메타데이터·서버 컴포넌트가 필요한 영역은 App Router를 씁니다. myib에서는 generateMetadata API로 단체·모금·봉사·캠페인 등 도메인별 layout 10개 영역에 SSR 메타데이터와 Open Graph를 적용했습니다. 반대로 이미 Pages Router로 안정적으로 도는 자산은 무리해서 옮기지 않고 유지했고요. 로폼처럼 AI가 문서를 점진적으로 생성하는 UX라면 React Server Component와 스트리밍이 사용자 체감 속도에 직접 도움이 된다고 봅니다."

### Q2. 디자인 시스템을 0→1로 어떻게 세우나요?
> "top-down으로 새 시스템을 받아오기 전에, 코드 안의 실제 사용처를 카테고리별로 추출하는 bottom-up으로 시작합니다. myib 입사 때 hex가 수백 곳, native alert가 58개, 어드민 테이블이 6종 비표준이었는데, 이걸 banner·button·icon·selector·table·toast 등 20여 개 카테고리로 토큰화하고 Storybook을 GitHub Pages에 자동 배포되게 CI를 세팅했습니다. 디자이너와 매번 합의하지 않아도 되도록 Figma와 코드 매핑을 DESIGN.md 단일 진실 문서로 묶었고요. 구체적으로는 아이콘 100여 개를 60개로 통합하고, FillButton 단일 표준으로 90개 파일을 일괄 교체했습니다."

### Q3. Radix UI는 써보셨나요? (← 정직하게)
> "프로덕션에 Radix를 직접 도입한 경험은 없습니다. 다만 제가 디자인 시스템을 만들 때의 철학이 Radix와 같습니다 — 키보드 포커스나 ARIA 같은 동작·접근성은 컴포넌트가 책임지고, 스타일은 토큰으로 분리하는 방식이요. 저는 평소에도 div에 ARIA를 붙이기 전에 button·dialog·nav 같은 semantic HTML을 먼저 선택하는 원칙으로 일해서, headless 프리미티브에 Tailwind 토큰을 입히는 Radix 패턴은 빠르게 적응할 수 있습니다. asChild나 Portal 같은 핵심 API는 미리 익혀두고 있습니다."

### Q4. (핵심) AI가 생성한 법률문서의 정확성·할루시네이션은 프론트엔드에서 어떻게 다뤄야 할까요?
> "법률문서는 한 줄 오타가 권리·계약에 직결되니까, 프론트엔드의 역할은 'AI 출력을 사용자가 안전하게 검증하도록 설계하는 것'이라고 봅니다. 네 가지로 접근합니다. 첫째, AI 출력은 항상 사용자가 검토·수정·재요청할 수 있는 편집 가능한 초안으로 제시합니다 — myib 모금 자동채우기에서 다룬 패턴입니다. 둘째, 확정 전에 명시적 검토 단계와 변경 diff를 보여줘서 사용자가 무엇이 바뀌는지 알게 합니다. 셋째, 생성 중·실패·빈 상태 세 가지를 모두 처리해서 스트리밍이 끊겨도 사용자가 혼란스럽지 않게 합니다. 넷째, 근거가 된 법령이나 판례를 표기해서 신뢰 신호를 줍니다. 로폼이 스케일업 팁스에서 'AI 법률문서 위험성 탐지'를 과제로 잡으신 걸 봤는데, 제가 AI 출력을 무비판적으로 수용하지 않고 교차 검증 단계를 거치는 작업 방식과 같은 결이라 흥미로웠습니다."

### Q5. 리치 텍스트 에디터와 LLM 출력을 어떻게 연결하나요? (Chat GLD 류)
> "myib에서 CKEditor 5 본문 에디터를 운영·커스터마이징하면서 콘텐츠 모델, 플러그인 구조, 이미지 업로드 파이프라인을 다뤘습니다. LLM과 에디터를 붙일 때 진짜 어려운 지점은 토큰 스트리밍을 에디터의 문서 모델에 안전하게 반영하는 부분이라고 봅니다 — 부분 업데이트를 어떻게 원자적으로 적용할지, 사용자가 동시에 편집할 때 충돌을 어떻게 막을지, 되돌리기(undo) 히스토리를 어떻게 깨지지 않게 유지할지요. 이 영역이 제가 가장 해보고 싶은 일이기도 합니다."

### Q6. 상태관리는 어떻게 가져가나요?
> "server state와 client state를 분리합니다. 서버 데이터는 Apollo Client로, 클라이언트 UI 상태는 Zustand와 Jotai를 도메인별로 나눠서 운영했습니다. Apollo는 v3에서 v4로 마이그레이션을 단독으로 했는데, MockedProvider 사양 변경과 타입 캐스팅 이슈를 전수 정리했습니다. 함께 백엔드 담당자와 deprecated된 GraphQL 쿼리·필드 10건을 식별해서 신규 스키마로 안전하게 이관했고요. React Query는 직접 안 써봤지만 server state 라이브러리라는 점에서 Apollo와 같은 카테고리라 빠르게 진입할 수 있습니다."

### Q7. 성능이나 접근성은 어떻게 챙기나요?
> "근본 원인을 고치는 쪽을 택합니다. 예를 들어 컴포넌트 배치 오차가 났을 때 사용처마다 스타일 래퍼로 봉합하는 대신 SVG의 viewBox를 24×24로 통일해서 한 번에 해결했습니다. 번들 측면에서는 미사용 컴포넌트·유틸·정적 파일 320여 개를 정리해서 빌드 비용을 줄였고요. 접근성은 semantic HTML을 기본값으로 두고, 콘솔 에러는 버그로 취급해서 통과 기준으로 봅니다."

### Q8. 테스트는 어떻게 하시나요? (← 정직하게)
> "Jest나 Vitest로 테스트 코드를 직접 작성한 경험은 많지 않습니다. 다만 검증 자체는 워크플로우에 내재화돼 있어서, Apollo MockedProvider로 데이터 레이어를 모킹하고 Playwright와 chrome-devtools MCP로 실제 브라우저에서 콘솔 에러 0과 모바일·데스크톱 렌더를 동시에 확인하는 방식으로 일합니다. 테스트 코드 직접 작성은 합류 후 빠르게 보완할 영역으로 보고 있습니다."

## C. 인성·컬처핏 면접

### Q9. 왜 이직하시나요? 왜 로폼인가요?
> "myib에서 4개 프로덕트를 넓게 보면서 책임 범위를 익혔는데, 이제 한 영역을 더 깊게 파고드는 환경으로 가고 싶었습니다. 제가 가장 잘하고 좋아하는 게 디자인 시스템과 AI 문서 인터페이스인데, 로폼은 이 두 축이 곁가지가 아니라 제품의 핵심이라는 점이 결정적이었습니다. 넓게 벌여온 경험을 한 곳에 집중해서 더 단단하게 만들고 싶습니다."

### Q10. 1인으로 일하셨는데, 팀에 합류해서 협업 잘하실 수 있나요?
> "오히려 1인이었기 때문에 협업이 일상이었습니다. 디자이너·백엔드·기획·마케팅과 매일 싱크를 맞춰야 했고, 서로 언어가 달라 생기는 오류를 줄이려고 DESIGN.md 같은 단일 진실 문서나 PRD 엔지니어링 주석 같은 장치를 만들었습니다. 다만 코드 리뷰를 주고받는 동료 개발자가 없던 환경이라, 팀에 합류하면 첫 과제는 제 주장을 앞세우기보다 기존 코드 문화와 리뷰 방식을 먼저 흡수하는 것이라고 생각합니다."

### Q11. 본인의 단점은?
> "시스템 전체의 큰 그림을 먼저 그리려는 성향이라, 새 코드베이스에 들어갈 때 머릿속 그림이 잡히기 전까진 첫 손이 느린 편입니다. 이걸 보완하려고 '정돈은 점진적으로, 0→1 신규 셋업은 의도적으로 빠르게' 두 모드를 명확히 구분해서 일합니다. 실제로 B2B 파트너 셋업 때는 신중함보다 추진력이 필요한 시점이라 판단해서 단기간에 캠페인·프로필·계좌 도메인까지 출시했습니다."

### Q12. 의견 충돌이 있었던 경험과 해결 방식은?
> "단체 상세 페이지 리디자인 때 '어떤 화면을 먼저 손볼지'를 두고 의견이 갈렸습니다. 저는 주관적 주장 대신 CS에 누적된 사용자 문의를 출발점으로 '신뢰 신호 분산·CTA 휘발성·임팩트 추상' 세 문제를 정의하고, North Star와 7개 하위 KPI, 그리고 클릭 가능한 인터랙티브 프로토타입을 만들어서 동료들이 가설을 직접 눌러보며 검토하게 했습니다. 말로 설득하기보다 작동하는 것과 데이터로 합의를 만드는 편입니다."

## D. 대표·임원 면접

### Q13. 로폼의 비전에 공감하시나요? 어떤 점에서?
> "'누구나 법률서비스를 누릴 수 있는 세상'이라는 방향에 공감합니다. 법률은 진입 장벽이 높아서 보통 사람은 계약서 한 장 앞에서도 막막한데, 그걸 누구나 완성할 수 있게 만드는 일은 화면만 그리는 게 아니라 사용자가 '이 한 장을 끝낼 수 있다'는 자신감을 설계하는 일이라고 봅니다. 제가 기부 도메인에서 신뢰가 흔들리면 사용자가 바로 떠나는 걸 매일 겪었기 때문에, 법률처럼 한 줄 오타가 권리에 직결되는 도메인의 무게를 가볍게 보지 않습니다."

### Q14. 법률 도메인은 처음이신데 괜찮을까요?
> "법률 도메인은 처음입니다. 하지만 myib에서 다룬 기부·결제·신원 인증은 한 번의 오류가 사용자의 자산과 신뢰로 직결된다는 점에서 법률문서와 결이 같습니다. 후원금 오결제 한 번에 사용자가 영영 떠나는 걸 코드로 겪었기 때문에, 정합성과 안정성이 곧 신뢰의 근거가 되는 도메인에서 어떻게 일해야 하는지는 체득돼 있습니다. 법률 지식 자체는 빠르게 배우겠지만, '신뢰가 제품의 전부인 도메인'을 다루는 태도는 이미 갖췄다고 생각합니다."

### Q15. 회사가 왜 당신을 뽑아야 하나요?
> "세 가지입니다. 첫째, 로폼이 찾는 Next.js B2B·Radix와 Tailwind 디자인 시스템·AI 기능 세 축이 제가 myib에서 실제로 해온 일과 거의 그대로 겹칩니다. 둘째, 저는 화면만 받아서 그리는 게 아니라 PRD를 직접 쓰고 그 위에서 구현하는 사람이라, 빠르게 크는 스타트업에서 기획과 구현 사이의 비용을 줄일 수 있습니다. 셋째, AI 출력을 무비판적으로 받지 않고 교차 검증으로 리스크를 통제하는 작업 방식이, 로폼이 스케일업 팁스에서 잡으신 '법률문서 위험성 탐지' 방향과 정확히 맞닿아 있습니다."

### Q16. 입사 후 1년, 무엇을 하고 싶으신가요?
> "첫 단계는 로폼의 React·Next.js 코드베이스를 빠르게 읽고, Radix와 Tailwind 기반 디자인 시스템·컴포넌트 라이브러리에 일관된 토대를 만드는 일입니다. myib에서 3개 프로덕트의 인터페이스를 단일 토큰 구조로 수렴시킨 경험을 그대로 옮길 수 있다고 봅니다. 그 토대 위에서 Chat GLD 같은 AI 문서생성 인터페이스의 사용자 경험을 함께 설계·구현하는 일에 도전하고 싶습니다 — 에디터 콘텐츠 모델과 LLM 출력 흐름을 잇는 영역이 제가 가장 하고 싶은 일입니다."

## F. 마인드셋 (당일 아침)

1. 과장하지 말 것 — 모르는 건 "직접 경험은 없지만 같은 카테고리라 빠르게 진입 가능"으로 정직하게.
2. 모든 답을 myib의 **구체 수치·사례**로 착지시킬 것 (2,200 커밋 / 320개 정리 / 아이콘 40% / 테이블 22-of-26 / Sentry 23건).
3. 로폼의 무게(법률 = 신뢰)를 진심으로 이해하고 있음을 톤으로 전할 것.
4. 역질문으로 '같이 일할 동료'의 자세를 보일 것.
