## 3. 진짜 리스크 & 정직한 답변 스크립트

### ① 경력 연수 (2년 4개월) — FE 공고가 "리더급"
- 솔직 인정 + 실무 밀도로 보완. **과장 금지.**
- 스크립트: *"경력 연수는 4년차 리더급에는 못 미칩니다. 다만 1인 프론트엔드로 B2C 앱·B2B 파트너·어드민·트랜잭션 메일 4개 프로덕트를 단독 운영하며 4개 레포 2,200+ 커밋을 쌓았고, 디자인 시스템 0→1·B2B 셋업·마이그레이션을 의사결정부터 끝까지 책임졌습니다. 리더 역할이 '직접 다 설계하고 끝맺는 것'이라면 그 밀도는 충분히 증명했다고 봅니다. 다만 팀을 이끄는 리더십은 아직 배우는 단계라 정직히 말씀드립니다."*
- ※ 만약 지원한 포지션이 리더급이 아닌 일반 FE라면 이 리스크는 약함 — 면접 초반에 포지션 레벨을 자연스럽게 확인.

### ② Radix UI 직접 경험 없음
- 카테고리가 같다는 점을 정확히 짚기. (Radix = headless/unstyled + a11y 내장 프리미티브)
- 스크립트: *"Radix를 프로덕션에 직접 쓴 경험은 없습니다. 다만 제가 디자인 시스템을 만들 때 택한 방향이 Radix 철학과 같습니다 — 동작·접근성(키보드 포커스·ARIA)을 컴포넌트가 책임지고, 스타일은 토큰으로 분리. 저는 semantic HTML을 먼저 두고 부족할 때만 ARIA로 보완하는 원칙으로 일해서, headless 프리미티브에 Tailwind 토큰을 입히는 패턴은 빠르게 적응할 수 있습니다."*
- 보강: Radix 핵심 개념(asChild, Portal, 접근성 기본 제공, controlled/uncontrolled) 면접 전 30분만 훑어두기.

### ③ 법률 도메인 첫 경험
- 자소서 논리 그대로: 신뢰가 무너지면 사용자가 떠나는 도메인이라는 점에서 결이 같다.
- 스크립트: *"법률 도메인은 처음입니다. 하지만 myib에서 다룬 기부·결제·신원 인증은, 한 번의 오류가 사용자의 자산·신뢰로 직결된다는 점에서 법률문서와 결이 같습니다. 후원금 오결제 한 번에 사용자가 떠나는 걸 코드로 겪었기 때문에, '한 줄 오타가 권리·계약에 영향을 주는' 로폼의 무게를 체감으로 이해합니다."*

### ④ 그 외 미스매치 대비 (물어보면)
- **테스트 코드(Jest/Vitest/Playwright 직접 작성)**: Apollo MockedProvider + Playwright·chrome-devtools MCP 검증 워크플로우 경험으로 우회. *"테스트 코드 직접 작성은 학습 후 진입 가능, 검증 자체는 워크플로우에 이미 내재화."*
- **모노레포/Turborepo**: 멀티 레포 자산 호환 경험으로 *"동일 패턴, 빠르게 적응"*.

## 4. 예상 기술 질문 & 답변 뼈대

1. **App Router vs Pages Router, 언제 무엇?**
   → SSR/메타데이터·서버 컴포넌트 필요한 신규는 App Router, 기존 자산은 Pages 유지. myib에서 generateMetadata로 도메인별 layout 10개 영역 SSR 메타데이터 적용. RSC/스트리밍은 AI 문서생성 UX(점진 렌더)와도 맞물림.

2. **디자인 시스템을 0→1로 어떻게 세우나? 토큰 거버넌스는?**
   → bottom-up(코드 실제 사용처 추출) → 카테고리 토큰화 → Storybook+CI → DESIGN.md 단일 진실 문서. hex 수백 개 치환, 아이콘 100→60 통합, FillButton 90파일 일괄 교체 사례.

3. **AI가 생성한 법률문서의 정확성/할루시네이션을 프론트에서 어떻게 다루나?** (이 회사 핵심 질문일 확률 높음)
   → ① AI 출력은 항상 사용자가 검토·수정·재요청할 수 있는 인터페이스(모금 자동채우기 경험), ② 확정 전 명시적 검토 단계·diff·근거 표시, ③ 스트리밍 중/실패/빈 상태 3-state 처리, ④ 출처(법령·판례) 표기로 신뢰 신호. 스케일업 팁스 '위험성 탐지'와 연결.

4. **리치 텍스트 에디터 + LLM 출력 연결 (Chat GLD 류)**
   → CKEditor 5 콘텐츠 모델·플러그인·이미지 파이프라인 경험. LLM 토큰 스트리밍을 에디터 문서 모델에 안전하게 반영(부분 업데이트·되돌리기·사용자 편집 충돌 방지)이 핵심 난제라고 봄.

5. **상태관리 선택 기준**
   → server state(Apollo/React Query)와 client state(Zustand/Jotai) 분리. 도메인별 store 분리 운영. Apollo v3→v4 마이그레이션(MockedProvider 사양 변경·타입 캐스팅) 단독 수행.

6. **성능/접근성**
   → semantic HTML 우선, 번들·미사용 코드 320여 개 정리로 빌드 비용 절감, viewBox 통일 같은 root-cause 방식. 접근성 빠뜨림은 critical로 본다.

## 5. 예상 인성·컬처 질문 & 답변 방향

1. **왜 이직? 왜 로폼?** → 4개 프로덕트 넓게 본 경험을 한 영역(디자인 시스템 + AI 문서 인터페이스)에 깊게 적용하고 싶다. 로폼은 그 두 축이 핵심.
2. **1인 운영 → 팀 합류, 협업 잘할 수 있나?** → 오히려 1인이라 모든 직군과 싱크 맞추는 게 일상이었다(DESIGN.md, PRD 주석). 합류 후 첫 과제는 기존 코드 문화 흡수.
3. **단점** → 큰 그림 먼저 보려 해서 진입 초반이 느릴 수 있음 → '정돈은 점진적, 0→1은 의도적으로 빠르게' 두 모드 분리로 보완.
4. **갈등 경험** → 의견 대립 시 데이터 지표·퍼널 다이어그램·작동하는 프로토타입으로 합의 유도(단체 상세 PRD 사례).
5. **연봉** → 자소서 기준 희망 4,500만(협의 가능). 최종연봉 3,425만에서의 상승 근거(책임 범위·기여)를 1줄로.
