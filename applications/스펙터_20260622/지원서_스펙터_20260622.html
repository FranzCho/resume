SELF-INTRODUCTION · 자기소개서
01
지원동기

지난 2년 5개월간 ㈜마이아이비에서 기부·모금·봉사 플랫폼의 프론트엔드를 담당하며, 신뢰가 흔들리는 순간 사용자가 바로 떠난다는 사실을 매일 코드로 마주했습니다. 후원금이 한 번 잘못 결제되면 사용자는 다시 돌아오지 않고, 단체가 한 번 부정확한 정보로 노출되면 플랫폼 전체에 대한 신뢰가 영향을 받습니다. 그래서 단체 상세 페이지 PRD 작성부터 디자인 시스템 토큰화, Sentry 누적 에러 전수 조사까지 화면 한 장의 가설을 정의하는 일부터 그 화면이 안정적으로 돌아가게 만드는 일까지를 한 사람이 이어서 책임지는 방식으로 일해 왔습니다.

스펙터를 지원한 이유는 "대한민국 채용 시장의 신뢰 자본을 데이터 사이언스로 다시 정의한다"는 회사의 방향이 제가 myib에서 익힌 일하는 방식과 맞닿는 부분이 많기 때문입니다. 채용은 한 줄의 잘못된 평가, 한 번의 잘못된 매칭이 지원자의 커리어 전체에 영향을 주는 영역입니다. 6,800여 개 대기업이 매일 사용하는 스펙터 코어와 AI 프로덕트(TEO, Effy)처럼 사용자의 평가·매칭 데이터가 곧 신뢰의 근거가 되는 영역에서는, 제품 표면의 안정성과 데이터 정합성이 그대로 사용자 신뢰의 근거가 된다고 생각합니다. 신뢰가 무너지는 순간 사용자가 떠나는 임팩트 도메인에서 쌓은 신뢰 중심의 프론트엔드 아키텍처 설계 및 운영 경험을 바탕으로, 채용 데이터 사이언스 영역에서 스펙터의 제품 가치를 함께 고도화하고자 지원했습니다.

02
직무 경험 ① — 디자인 시스템을 코드에서 끌어올린 1년

myib 입사 시 코드베이스는 hex 컬러가 수백 곳에 흩어져 있고, native alert() 호출이 58개, 어드민 테이블은 6종이 비표준으로 운영되고 있었습니다. 단순히 컴포넌트를 새로 만드는 것이 아니라, 기존 코드에 이미 형성된 패턴을 정확히 읽어내는 것에서 시작해야 한다고 판단했습니다. 디자이너에게 "top-down으로 새 시스템을 받아오자"고 하기 전에, 코드 안의 실제 사용처를 카테고리별로 추출하는 bottom-up 접근을 택했습니다.

이 원칙으로 약 1년에 걸쳐 banner·button·icon·selector·table·toast 등 20여 개 카테고리를 토큰화하고, Storybook을 구축해 GitHub Pages에 자동 배포되도록 CI를 세팅했습니다. 디자이너·개발자가 매번 새로 합의하지 않아도 되도록 Figma↔코드 매핑을 단일 진실 문서(DESIGN.md)로 도입했습니다. 파트너웹 디자인 시스템은 DESIGN-AUDIT → 토큰 신설 → primitives → 일괄 이관 4단계로 정비하면서 arbitrary 값 340여 파일을 토큰 기반 구조로 옮기고, AA 색 대비 97% 준수와 lint:tokens 0 에러를 달성해 토큰 거버넌스를 정립했습니다. 어드민 영역에서는 DataTable·StatusBadge를 신규 구축한 뒤 admin2의 26개 테이블 중 22개를 표준 컴포넌트로 교체했고, 흩어져 있던 native alert 58건은 일관된 showToast 패턴으로 일괄 전환하면서 사용되지 않던 useMessage 및 MessageContext 레이어까지 함께 제거했습니다.

이와 병행해 레거시 admin 디렉토리 190개 파일을 외부 의존성과 함께 단일 commit으로 이관·정리했고, 하루 동안 미사용 컴포넌트·유틸·public 정적 파일 약 320여 개를 일괄 삭제했습니다. 아이콘 영역에서는 카테고리별 100여 개를 약 60개로 통합(40% 절감)하면서 PNG 16종을 SVG 컴포넌트로 전환했고, 버튼은 FillButton 단일 표준으로 통일하며 90개 파일을 한 번에 교체해 코드베이스의 부피와 빌드 비용을 함께 줄였습니다.

이 작업은 화려한 신규 기능은 아니었지만, 결과적으로 B2C 앱·B2B 파트너·어드민 3개 프로덕트가 같은 토큰과 컴포넌트로 수렴하면서 신규 화면 하나를 만드는 데 드는 의사결정 비용이 줄었습니다. 스펙터가 우대하는 "Radix UI + Tailwind CSS 기반 컴포넌트 제작"과 모노레포 환경의 토큰 거버넌스에 그대로 이식할 수 있는 토대라고 생각합니다.

03
직무 경험 ② — 기획서를 직접 쓰고 그 위에서 코드를 짠다

프론트엔드 개발자로 일해왔지만, 화면 한 장을 만들 때마다 사용자가 어디서 들어와 무엇을 결정하고 어디로 빠져나가는지를 먼저 그려보는 사람입니다. 기능 명세를 받아 구현하기보다, 유저의 의사결정 동선·고객사(단체·기업)의 노출 요구·웹에서의 표현 방식 세 축을 한 문서에서 함께 설계한 뒤 자기 손으로 구현하는 사이클로 일합니다. 의견이 갈리는 상황에서도 화면 한 장이 놓일 시스템 전체에서 무엇이 맞춰져야 하는지를 함께 보고 하나씩 합의를 만들어가는 방식이, 화면이 제 기능을 하도록 만든다고 믿습니다. 스펙터가 명시한 "기획서대로 화면만 그리는 코더가 아닌 UX Architect"라는 인재상은, 제가 가장 깊이 키워 온 역량과 그대로 겹칩니다.

기획서의 입력은 고객지원 채널에 누적된 사용자 문의, 진입 경로별 동선 분석, 백엔드·운영·마케팅 담당자와의 직접 협의 세 채널에서 옵니다. 대표 사례인 단체 상세 페이지 리디자인 PRD(v2.0)에서는 "어떤 단체인지 신뢰가 안 간다", "기부 버튼이 어디 있는지 모르겠다" 같은 CS 응대 기록을 출발점으로 '신뢰 신호 분산 / CTA 휘발성 / 임팩트 추상' 세 가지 문제를 정의했고, 화면 단위가 아니라 진입, 검토, 결정, 결제로 이어지는 퍼널(Funnel) 전체에서 North Star와 7가지 하위 KPI(Sub-KPI)를 잡았습니다. 페르소나를 세 그룹으로 세분화하고, 4단계 결제 흐름을 Mermaid 플로우 다이어그램과 7개 화면 인터랙티브 프로토타입으로 옮겨 동료들이 클릭만으로 가설을 검토할 수 있도록 했습니다. 결제 전환 흐름도 직접 정비해 CheckoutSuccess 화면을 409줄 → 134줄로 축소하며 정기/일시 카피 분기와 "매월 N일 결제 예정" 부가 정보를 신설했고, 마이페이지 기획서에서는 9개 메뉴 × 5개 상태 케이스(약 45 조합)를, 모금 상세 리디자인에서는 100여 가지 케이스를 같은 방식으로 정리했습니다.

동일한 책임 범위를 바탕으로 B2B 파트너 프론트엔드 시스템을 Next.js 기반으로 제로 베이스에서 단독으로 셋업하며 크로스 기능 조직 간 도메인 인터페이스 설계 합의를 주도했습니다. Apollo Client v4 마이그레이션 시 MockedProvider 사양 변경과 타입 캐스팅 이슈를 해결하고, 서버 장애로 직결될 수 있는 deprecated GraphQL 쿼리 10건을 신규 스키마로 안전하게 이관했으며, Sentry 누적 에러 23건의 전수 조사와 예외 처리 보완까지 완수해 화면 위의 가설부터 하부 데이터·관측성 레이어의 정합성까지 전 과정을 책임지는 엔지니어링 역량을 증명했습니다. 검색 노출 측면에서도 Next.js App Router의 generateMetadata API를 활용해 단체·모금·봉사·캠페인 등 도메인별 메타데이터와 Open Graph 정보를 일관된 SSR 구조로 정리했습니다.

04
성격과 일하는 방식 — 정밀함·완결성과 협업, AI 레버리지

성격의 가장 큰 특징은 모호한 현상에 머무르지 않고 근본적인 원인을 추적하여 체계화하는 정밀함과 완결성입니다. 컴포넌트 배치 오차를 스타일 래퍼로 임시 봉합하는 대신 SVG 자체의 viewBox를 통일하여 해결하는 것처럼, 한 번 정립한 구조적 원칙을 코드베이스 전체에 일관되게 적용하려 합니다. 예외 처리 미비로 발생할 수 있는 잠재 장애를 막기 위해 센트리 에러 23건을 전수 조사하고 하루 만에 해결한 것 또한 이러한 완결성 중심의 성격에서 비롯되었습니다. 렌더링 속도 0.1초의 차이를 임시방편이 아니라 구조로 풀어내는 일에 가장 큰 보람을 느끼는 성격입니다.

이러한 정밀한 성격은 협업을 할 때 말로 앞서기보다 작동하는 시스템과 정제된 문장으로 소통하는 방식으로 발현됩니다. 1인 프론트엔드 개발자로서 다양한 직군과 협업하며, 서로의 언어가 달라 발생하는 싱크 오류를 줄이는 데 집중해 왔습니다. 디자이너와의 커뮤니케이션 비용을 줄이기 위해 피그마와 코드를 일원화한 단일 진실 문서를 도입하고, 백엔드 개발자와의 데이터 규격을 맞추기 위해 PRD에 엔지니어링 주석을 달아 사전에 합의하는 방식이 저의 대표적인 협업 스타일입니다. 주관적인 의견 대립이 생기더라도 데이터 지표와 퍼널 다이어그램을 제시하며 전사적인 합의를 주도적으로 이끌어냅니다.

반면 시스템 전체의 구조와 큰 그림을 먼저 조망하려 하기에 초기 아키텍처 설계 단계에서 다소 신중해진다는 점이 있습니다. 이러한 신중함을 보완하고 비즈니스의 속도를 맞추기 위해 저는 두 가지 레버리지를 활용합니다.

첫째는 고속 실행이 필요한 타이밍에 의도적으로 업무 모드를 전환하는 메타인지이며, 둘째는 Claude·Gemini·Claude Code 같은 AI 도구를 가상 아키텍처 파트너로 워크플로우에 내재화하는 능력입니다. 단, 채용 데이터의 정확성과 사용자 신뢰가 최우선인 도메인 특성을 고려하여 AI가 제안한 구조나 데이터 치환 결과물을 무비판적으로 수용하지 않고, 저의 정밀한 성향을 바탕으로 철저한 교차 검증 단계를 거쳐 기술적 할루시네이션이나 결함 리스크를 통제합니다. 이러한 기술 제어를 통해 신중함이 만드는 속도의 제약을 생산성 저해 없이 극복하고 있으며, "시장의 룰을 부수는 구간에서는 어떻게든 일이 되게 만들어야 한다"는 스펙터의 일하는 방식과 결을 같이 한다고 생각합니다.

05
입사 후 포부

스펙터에서 함께 일할 기회를 주신다면, 제 주장을 앞세우기보다 가장 먼저 팀의 소통 방식과 기존 코드베이스의 문화를 온전히 흡수하는 데 집중하겠습니다. 새로운 조직에 합류할 때 가장 중요한 것은 동료들에게 누를 끼치지 않고, 기존 시스템의 안정적인 운영에 빠르게 보탬이 되는 것이라 생각합니다. 스펙터 코어와 TEO·Effy 등 6,800여 개 대기업이 매일 사용하는 채용 데이터·AI 핵심 서비스에 성실히 적응하여, 기존 팀원분들이 쌓아 올린 견고한 토대 위에 묵묵히 제 몫의 개발 리소스를 더하겠습니다. 채용 데이터 사이언스 도메인의 특성상 평가·매칭 데이터의 정확성과 사용자 신뢰가 직결되는 만큼, 자만하지 않고 기초부터 철저히 배우며 팀의 일원으로서 완벽히 녹아들겠습니다.

이후 비즈니스 방향성에 맞추어 제가 가진 경험들이 팀원분들의 업무 부담을 실질적으로 덜어주는 무기가 되기를 바랍니다. 마이아이비에서 제로 베이스부터 단독으로 시스템을 구축하고 요구사항 문서(PRD)를 주도적으로 다듬어본 경험은, 미국 플립과 글로벌 스케일업으로 확장하며 채용 시장의 패러다임 전환을 만들어 가는 단계에서 그대로 옮겨갈 수 있는 자산이 될 것입니다. 특히 myib에서 20여 개 카테고리 토큰화부터 시작해 B2C·B2B·어드민 3개 프로덕트의 인터페이스를 단일 토큰 구조로 수렴시키고, client state는 Zustand 14개 store로, server state는 Apollo Client v3→v4 마이그레이션을 단독 전담하며 분리 운영한 경험은, 스펙터의 Radix UI + Tailwind CSS + Turborepo 컴포넌트 생태계와 TanStack Query + Zustand 상태관리 패턴에 직접 연결되는 부분이라고 생각합니다. 청소년기 뉴질랜드 거주로 다져진 비즈니스 영어 능력 역시 글로벌 스케일업 단계에서 영문 인터페이스·문서·협업 어디서든 즉시 활용할 수 있는 부분입니다. 또한 평소 연구해 온 AI 협업 워크플로우 역시 혼자만의 생산성에 머무르지 않고, 팀의 전체적인 개발 손길을 가볍게 만들고 동료들과 다 함께 성장하는 데 조용히 기여하고 싶습니다.

"대한민국 채용 시장의 신뢰 자본을 데이터 사이언스로 다시 정의한다"는 스펙터의 방향, 그리고 채용 시장의 패러다임 전환은 결코 혼자서 만들 수 없다고 믿습니다. 시계의 바늘보다 프론트엔드의 렌더링 속도가 0.1초 빨라질 때 더 큰 아드레날린을 느낀다는 공고의 한 줄에 깊이 공감하며, 팀원들과 긴밀히 협업하고 끊임없이 배우며 스펙터가 지향하는 가치를 사용자가 마주하는 화면에서 완벽한 신뢰성으로 구현해 나가는 가장 성실하고 든든한 동료가 되겠습니다.
