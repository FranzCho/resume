## 1. 나의 업무 스타일 및 장점과 단점

저의 가장 큰 장점은 **모호한 현상에 머무르지 않고 근본적인 원인을 추적해 체계화하는 정밀함과 완결성**입니다. 컴포넌트 배치 오차를 사용처에서 스타일 래퍼로 임시 봉합하는 대신 SVG 자체의 viewBox를 24×24로 통일해 해결하는 것처럼, 한 번 정립한 구조적 원칙을 코드베이스 전체에 일관되게 적용하려 합니다. myib에서 예외 처리 미비로 자라고 있던 Sentry 누적 에러 23건을 하루 만에 전수 조사하고 정리한 것, 그리고 아이콘 100여 개를 60개로 통합(40% 절감)하면서 PNG 16종을 SVG 컴포넌트로 일괄 전환한 것도 같은 성향에서 비롯되었습니다. **렌더링 속도 0.1초의 차이를 임시방편이 아니라 구조로 풀어내는 일에 가장 큰 보람을 느끼는 성격**입니다.

이러한 정밀한 성격은 협업에서 **말로 앞서기보다 작동하는 시스템과 정제된 문장으로 소통하는 방식**으로 발현됩니다. 1인 프론트엔드 개발자로서 디자이너·백엔드·기획·마케팅 등 다양한 직군과 협업하며, 서로의 언어가 달라 발생하는 싱크 오류를 줄이는 데 집중해 왔습니다. 디자이너와의 커뮤니케이션 비용을 줄이기 위해 Figma와 코드를 일원화한 단일 진실 문서(DESIGN.md)를 도입하고, 백엔드 개발자와의 데이터 규격을 맞추기 위해 PRD에 엔지니어링 주석을 달아 사전에 합의하는 방식을 자리잡게 했습니다. 의견이 갈리는 상황에서도 데이터 지표와 퍼널 다이어그램을 제시하며 전사적 합의를 주도적으로 이끌어내려 합니다.

또 하나의 핵심 작업 방식은 **AI 도구(Claude, Gemini, Claude Code)를 가상 아키텍처 파트너로 워크플로우에 내재화**하는 능력입니다. 기획서 초안 단계에서 사용자 동선과 페르소나를 빠르게 정리하고, 디자인 시스템 작업에서는 수백 곳에 흩어진 hex 치환을 일괄 처리하며, Apollo Client v4 마이그레이션 같은 대규모 타입 치환을 가속화하는 데 활용했습니다. 단, AI가 제안한 구조나 데이터 치환 결과물을 무비판적으로 수용하지 않고 교차 검증 단계를 거쳐 기술적 할루시네이션과 결함 리스크를 통제합니다.

**단점은 시스템 전체의 구조와 큰 그림을 먼저 조망하려는 성향**에서 옵니다. 새로운 코드베이스에 들어갈 때 머릿속에 큰 그림이 그려지기 전까지 첫 손이 늦은 편이어서, 진입 단계의 속도가 더디게 느껴질 수 있습니다. 이를 보완하기 위해 *'정돈은 점진적으로, 0→1 신규 셋업은 의도적으로 빠르게'* 두 가지 모드를 명확히 구분해 일하고 있으며, myib에서 B2B 파트너 프론트엔드 셋업이 단기간에 캠페인·프로필·계좌 도메인까지 출시된 것은 이 시점에는 신중함보다 추진력이 필요하다는 판단이었습니다. **시장의 룰을 부수는 구간에서는 어떻게든 일이 되게 만들어야 한다**는 스펙터의 일하는 방식과 결을 같이 한다고 생각합니다.

## 2. 내가 이 포지션에 적합한 사람인 이유

스펙터가 정의한 *"기획서대로 화면만 그리는 코더가 아닌 UX Architect, 비즈니스 임팩트 창출자"* 라는 인재상과, 채용공고에 명시된 5개 우대사항 *(모노레포·디자인 시스템·웹 퍼포먼스·WebSocket·AI/LLM 연동)* 중 4개 영역이 제가 myib에서 누적해 온 작업 범위와 직접 일치합니다.

**① B2B Enterprise SaaS 프론트엔드 0→1 단독 셋업 (주요업무 1번 직접 매칭)**

myib에서 B2B 파트너 프론트엔드 시스템을 Next.js 기반으로 제로베이스에서 단독 셋업했습니다. Tailwind CSS 컬러 토큰과 빌드 구성을 메인 웹앱 디자인 시스템과 결합해 멀티 레포지토리 간 자산 호환성을 확보했고, 단체 회원 가입·인증, 캠페인 등록·관리, 프로필 관리, 계좌·정산 도메인을 0→1로 출시했습니다. Next.js 미들웨어를 제어해 메인 앱의 세션 토큰 갱신 패턴을 파트너 시스템으로 안전하게 이식하고, 미인증 API 호출 가드·에러 기반 리다이렉트 분기까지 구축했습니다. **6,800여 개 대기업이 사용하는 스펙터 코어와 신규 AI 프로덕트의 아키텍처를 주도하는 자리에 그대로 옮길 수 있는 경험**이라고 봅니다.

**② 디자인 시스템 + Storybook + Tailwind CSS 직접 매칭 (우대사항)**

myib 입사 시 코드베이스는 hex 컬러가 수백 곳에 흩어져 있고, native alert() 호출이 58개, 어드민 테이블 6종이 비표준으로 운영되고 있었습니다. 약 1년에 걸쳐 banner·button·icon·selector·table·toast 등 20여 개 카테고리를 토큰화하고, **Storybook을 구축해 GitHub Pages에 자동 배포되도록 CI를 세팅**했습니다. 파트너웹 디자인 시스템은 DESIGN-AUDIT → 토큰 신설 → primitives → 일괄 이관 4단계로 정비하면서 arbitrary 값 340여 파일을 토큰 기반 구조로 옮기고, **AA 색 대비 97% 준수와 lint:tokens 0 에러를 달성**해 토큰 거버넌스를 정립했습니다. 공통 컴포넌트 DataTable·StatusBadge를 신규 설계해 내부 어드민(admin2)의 26개 테이블 중 22개를 표준 교체했고, FillButton 단일 표준화로 90개 파일을 일괄 교체했습니다. 스펙터의 *"Radix UI + Tailwind CSS 기반 디자인 시스템 컴포넌트 제작"* 영역에 그대로 옮길 수 있는 토대입니다.

**③ 웹 퍼포먼스·번들 사이즈 정돈 직접 매칭 (우대사항)**

스펙터가 우대하는 *"렌더링·번들 사이즈 등 웹 퍼포먼스 최적화"* 영역은 제가 가장 정량적으로 증명한 작업 카테고리입니다. 레거시 admin 디렉토리 190개 파일을 외부 의존성과 함께 단일 commit으로 이관하고, 미사용 컴포넌트·유틸·정적 파일 약 320여 개를 일괄 삭제하면서 코드베이스 부피와 빌드 시간을 동시에 줄였습니다. 아이콘은 100여 개를 60개로 통합(40% 절감)하고 PNG 16종을 SVG 컴포넌트로 전환했으며, 버튼 시스템 FillButton 단일 표준화로 90개 파일의 결합 구조를 일괄 교체했습니다. Sentry 모니터링은 4개 레포지토리 설정을 단일 표준 양식으로 통일하고 누적 에러 23건을 전수 조사해 글로벌 예외 처리 미비로 인한 잠재 크래시 리스크를 사전에 제거했습니다. **유저 불편을 초래하는 병목 구간을 데이터 기반으로 찾아내고 기술적으로 해결한다**는 주요업무 2번과 결을 같이 합니다.

**④ 상태관리 server/client 분리 + 데이터 레이어 + 관측성**

스펙터의 *"TanStack Query 서버 상태 관리 + Zustand 클라이언트 상태 관리"* 패턴은 제가 myib에서 운영해 온 구조와 카테고리가 동일합니다. **client state는 Zustand로 두 레포지토리 14개 store**(결제·시스템·사용자·기부·봉사·주민번호 입력·토큰)를 도메인별 분리해 운영했고, **server state는 Apollo Client v3→v4 마이그레이션을 단독으로 전담**하여 MockedProvider 사양 변경·타입 캐스팅을 전수 정리했습니다. 백엔드와 협업해 deprecated GraphQL 쿼리·필드 10건을 신규 스키마로 안전하게 이관하면서, 프론트엔드와 백엔드 경계를 유연하게 넘나드는 작업도 익숙해졌습니다. TanStack Query는 server state 라이브러리로서 같은 카테고리이기에 빠르게 진입 가능하다고 보고 있습니다.

**⑤ AI 기능 연동 + AI 협업 워크플로우 (우대사항)**

myib에서 모금 만들기 화면의 **LLM 기반 AI 자동채우기 기능을 개선·연동**하면서 AI가 출력한 콘텐츠를 사용자가 검토·수정·재요청할 수 있는 인터페이스 패턴을 다뤘습니다. 더 넓게는 평소 Claude·Gemini·Claude Code를 가상 아키텍처 파트너로 워크플로우에 내재화하여 대규모 레거시 타입 치환을 가속화하고, 교차 검증 단계를 거쳐 결함 리스크를 통제하는 작업 방식을 정착시켰습니다. 스펙터의 **TEO, Effy 같은 AI 프로덕트의 실시간 인터랙션 UI를 설계·고도화**하는 영역과 직접 연결됩니다.

**⑥ PRD 직접 작성·구현 = UX Architect 인재상 매칭**

스펙터가 명시한 *"UX Architect"* 인재상은 제가 myib에서 가장 깊이 키워 온 역량입니다. 단체 상세 페이지 리디자인 PRD(v2.0)에서는 *"어떤 단체인지 신뢰가 안 간다", "기부 버튼이 어디 있는지 모르겠다"* 같은 CS 응대 기록을 출발점으로 *'신뢰 신호 분산 / CTA 휘발성 / 임팩트 추상'* 세 가지 문제를 정의했고, 화면 단위가 아니라 진입 → 검토 → 결정 → 결제로 이어지는 퍼널 전체에서 **North Star Metric과 7가지 하위 KPI(Sub-KPI)** 를 잡았습니다. 페르소나를 세 그룹으로 세분화하고, 4단계 결제 흐름을 Mermaid 다이어그램과 7개 화면 인터랙티브 프로토타입으로 옮겨 동료들이 클릭만으로 가설을 검토할 수 있게 했습니다. 결제 전환 흐름을 직접 정비해 CheckoutSuccess 화면을 409줄→134줄로 축소하면서 정기/일시 카피 분기와 *"매월 N일 결제 예정"* 부가 정보를 신설한 것은, *"비즈니스 임팩트 창출"* 인재상과 맞닿는 작업이라고 봅니다.

## 3. 나의 지원 동기 및 포부

지난 2년 5개월간 myib에서 기부·모금·봉사 플랫폼의 프론트엔드를 담당하며, **신뢰가 흔들리는 순간 사용자가 바로 떠난다**는 사실을 매일 코드로 마주했습니다. 후원금이 한 번 잘못 결제되면 사용자는 다시 돌아오지 않고, 단체가 한 번 부정확한 정보로 노출되면 플랫폼 전체에 대한 신뢰가 영향을 받습니다. 그래서 단체 상세 페이지 PRD 작성부터 디자인 시스템 토큰화, Sentry 누적 에러 전수 조사까지 화면 한 장의 가설을 정의하는 일부터 그 화면이 안정적으로 돌아가게 만드는 일까지를 한 사람이 이어서 책임지는 방식으로 일해 왔습니다.

스펙터를 지원한 이유는 *"대한민국 채용 시장의 신뢰 자본을 데이터 사이언스로 다시 정의한다"* 는 방향이 제가 myib에서 익힌 일하는 방식과 결을 같이 한다고 느꼈기 때문입니다. 채용은 한 줄의 잘못된 평가, 한 번의 잘못된 매칭이 지원자의 커리어 전체에 영향을 주는 영역입니다. 신뢰가 무너지는 순간 사용자가 떠나는 임팩트 도메인과 채용 도메인은, **제품 표면의 안정성과 데이터 정합성이 그대로 사용자 신뢰의 근거**가 된다는 점에서 매우 닮아 있습니다.

특히 스펙터가 명시한 *"기획서대로 화면만 그리는 코더는 사양합니다"* 라는 한 줄은 제가 일해 온 방식 그 자체였습니다. myib에서 기능 명세를 받아 구현하기보다, **유저의 의사결정 동선·고객사(단체·기업)의 노출 요구·웹에서의 표현 방식 세 축을 한 문서에서 함께 설계한 뒤 자기 손으로 구현하는 사이클**로 일해 왔습니다. 단체 상세 PRD의 North Star + 7개 Sub-KPI 설계, 캠페인 등록 PRD의 13개 사용성 피드백 반영, CheckoutSuccess 409줄→134줄 축소까지, 모두 *"이 버튼 여기 달아주세요"* 가 아니라 *"왜 이 버튼이 여기 있어야 하는가"* 를 먼저 묻고 시작한 작업이었습니다.

입사 후 첫 1년은 **스펙터 코어와 TEO·Effy의 코드베이스를 빠르게 읽어 들이고, Radix UI + Tailwind CSS + Turborepo 기반 컴포넌트 생태계의 토대를 두텁게 다지는 일**에 집중하고 싶습니다. myib에서 3개 프로덕트의 인터페이스를 단일 토큰 구조로 수렴시키고 admin2의 26개 테이블 중 22개를 표준 컴포넌트로 교체한 경험은, 폭발적 성장을 만들어내야 하는 스펙터의 컴포넌트 생태계에 그대로 이식할 수 있는 자산이라고 봅니다. Storybook 자동 배포 CI와 Sentry 단일 표준화 같은 운영 토대도 자연스럽게 함께 가져갈 수 있는 부분입니다.

그 토대 위에서 **TEO·Effy 같은 AI 프로덕트의 실시간 인터랙션 인터페이스를 설계·고도화하는 일**에 도전하고 싶습니다. AI 면접관과 인재 적합도 분석 엔진의 결과물을 *"단 1초의 렌더링 지연 없이 가장 우아하고 직관적인 UX로 번역"* 하는 자리는, 평소 AI 협업 워크플로우와 PRD 직접 작성을 함께 익혀온 작업 방식과 가장 맞닿아 있는 자리라고 생각합니다. 청소년기 뉴질랜드 거주로 다져진 비즈니스 영어 능력은, 미국 플립과 글로벌 스케일업 단계에서 영문 인터페이스·문서·협업 어디서든 즉시 활용할 수 있는 부분입니다.

스펙터가 그리는 *"채용 시장의 패러다임 전환"* 은 결코 혼자서 만들 수 없다고 믿습니다. **시계의 바늘보다 프론트엔드의 렌더링 속도가 0.1초 빨라질 때 더 큰 아드레날린을 느낀다**는 공고의 마지막 한 줄에 깊이 공감합니다. 적당히 타협하지 않는 최정예 원팀 속에서, 사용자가 마주하는 화면에서 스펙터의 압도적 기술력이 압도적 사용자 경험으로 번역되도록 가장 성실하고 든든하게 기여하겠습니다.
