SELF-INTRODUCTION · 자기소개서
01
지원동기

지난 2년 5개월간 ㈜마이아이비에서 기부·모금·봉사 플랫폼의 프론트엔드를 담당하며, 신뢰가 흔들리는 순간 사용자가 바로 떠난다는 사실을 매일 코드로 마주했습니다. 후원금이 한 번 잘못 결제되면 사용자는 다시 돌아오지 않고, 단체가 한 번 부정확한 정보로 노출되면 플랫폼 전체에 대한 신뢰가 영향을 받습니다. 그래서 단체 상세 페이지 PRD 작성부터 디자인 시스템 토큰화, Sentry 누적 에러 전수 조사까지 화면 한 장의 가설을 정의하는 일부터 그 화면이 안정적으로 돌아가게 만드는 일까지를 한 사람이 이어서 책임지는 방식으로 일해 왔습니다.

파트리지시스템즈를 지원한 이유는 AI 시대에 폭증하는 데이터를 가볍고 빠르게 처리한다는 방향과, 그 위에서 일하는 방식이 제가 지난 2년간 스스로 만들어 온 개발 환경과 정확히 겹치기 때문입니다. 자율주행·센서 데이터의 관제와 적재 콘솔처럼 화면 한 장이 곧 데이터 신뢰의 근거가 되는 영역에서는, 제가 임팩트 도메인에서 익힌 표면의 일관성과 데이터 정합성을 함께 책임지는 방식이 그대로 통한다고 생각합니다.

특히 공고의 「이렇게 일합니다」에 적힌 세 가지 — 규칙의 코드화, 자동 가드로 이어지는 검증, 계획·실행·검증·회고로 이어지는 정형화된 작업 흐름 — 은 제가 이미 회사 코드베이스에서 운영하고 있는 체계입니다. 도구에 열려 있는 정도가 아니라, 에이전트가 읽는 규칙 문서와 CI 가드를 직접 설계해 그 위에서 개발해 온 사람으로서 데이터반(databahn)의 프론트엔드를 함께 만들고자 지원했습니다.

02
직무 경험 ① — 여러 앱이 공유하는 디자인 시스템을 코드에서 끌어올리다

myib 입사 시 코드베이스는 hex 컬러가 수백 곳에 흩어져 있고, native alert() 호출이 58개, 어드민 테이블은 6종이 비표준으로 반복되고 있었습니다. 하나하나 고치는 대신 반복되는 패턴을 읽어내 표준 하나로 수렴시키는 접근을 택했습니다. alert 58건은 showToast 단일 패턴으로 일괄 전환하며 그 아래의 useMessage·MessageContext 레이어까지 제거했고, 비표준 테이블 6종은 DataTable·StatusBadge 두 컴포넌트로 추상화해 어드민 테이블 26개 중 22개를 교체했습니다. 화면마다 제각각이던 인라인 빈 상태 13곳과 스피너 7곳도 프리미티브 3종으로 흡수했습니다.

이 추상화를 개인 습관이 아니라 여러 앱이 공유하는 시스템으로 남기는 데 집중했습니다. banner·button·icon·selector·table·toast 등 20여 카테고리를 토큰화하고, Storybook 컴포넌트 플레이그라운드를 구축해 GitHub Pages로 자동 배포되는 CI 파이프라인을 세팅했으며, Figma↔코드 매핑을 단일 진실 문서(DESIGN.md)로 정의해 디자이너와 매번 새로 합의하지 않아도 되게 만들었습니다. 최근에는 B2C 앱과 B2B 파트너 두 레포의 타이포·radius·shadow·semantic 토큰 집합을 시각 변화 0을 전제로 완전히 일치시키고, baseline(ratchet) 방식 토큰 lint와 CI 게이트를 양쪽에 이식했습니다. prettier·TypeScript·Tailwind·ESLint 도구 체인 버전까지 수렴시키고 cross-repo 문서 동기화 훅을 배선해, 단일 레포 거버넌스를 멀티 레포 체계로 확장했습니다.

복잡한 요구사항을 단순한 구조로 되돌린 사례로는 결제 완료 화면이 있습니다. 정기/일시 결제·매칭·영수증 분기가 뒤엉켜 409줄이 된 화면을 상태 조합부터 다시 정의해 134줄의 단순한 구조로 재설계하면서, 헤딩 카피 분기와 「매월 N일 결제 예정」 같은 정보는 오히려 늘렸습니다. 임팩트는 B2C 앱·B2B 파트너·어드민 3개 프로덕트가 같은 언어로 수렴해 신규 화면 하나의 의사결정 비용이 줄어든 것이고, 러닝 포인트는 복잡도를 줄이는 것과 요구사항을 줄이는 것은 전혀 다른 일이라는 점, 그리고 좋은 추상화는 새 개념을 발명하는 게 아니라 코드에 이미 반복되고 있는 패턴을 정확히 읽어내는 데서 시작한다는 점입니다. 여러 앱이 사내 디자인 시스템을 함께 쓰는 모노레포 환경에서 바로 이어 쓸 수 있는 경험이라 생각합니다.

03
직무 경험 ② — 성능 문제를 계측으로 드러내고, 근본 원인까지 내려가다

성능은 체감이 아니라 숫자에서 출발해야 한다고 생각합니다. 아무도 요청하지 않았지만 프로덕션 홈을 직접 실측해 네트워크 페이로드 18.5MB, LCP 리소스 로드 지연 5,050ms라는 기준선을 만들고 P0~P3 백로그를 스스로 수립했습니다. 그 과정에서 앱 전체 SSR 마크업이 비어 있던 근본 원인이 세션 로딩 게이트에 있다는 것을 추적해 서버 렌더를 root HTML 0자 → 95KB로 복구했고, LCP 배너를 서버 페치로 전환, 폰트 2MB를 dynamic subset으로, 렌더 차단 체인 680ms를 제거했습니다. 같은 방법론을 파트너웹으로 확장하며 xlsx를 ESM 엔트리 지연 로딩으로 전환해 정산 화면의 초기 번들을 줄였는데, 이때 bare 지연 로딩이 SSR에서 require is not defined 500 오류를 내는 원인까지 진단해 해결했습니다. 번들 자체도 아이콘 100여 개를 60개로(40%↓) 통합하고 PNG 16종을 SVG 컴포넌트로 전환, 미사용 컴포넌트·유틸·정적 파일 약 320여 개를 회귀 없이 정리했습니다.

장애 대응과 원인 분석도 같은 방식입니다. SSR·서버 페치에서 산발적으로 발생하던 401 오류는 재시도로 덮지 않고 NextAuth jwt 콜백의 만료 토큰 갱신 누락이라는 근본 원인을 찾아 수정했고, refresh token rotation 과정에서 새 토큰 전파가 누락돼 생기던 조기 로그아웃도 원인부터 해결한 뒤 rememberMe 세션 쿠키 정규화 모듈을 TDD로 구현했습니다. 4개 레포의 Sentry 설정을 단일 표준으로 통일하고 누적 에러 23건을 전수 조사해 글로벌 예외 처리 미비로 인한 잠재 크래시를 사전에 제거했으며, 의존성 취약점은 63건 → 10건(critical/high 전부 제거), eslint error는 97건 → 0건으로 정리했습니다.

레거시를 새 기반으로 옮길 때도 빅뱅이 아니라 안전 경로를 먼저 설계합니다. Apollo Client v3→v4 마이그레이션을 전담하며 MockedProvider 사양 변경과 타입 캐스팅 이슈를 추적해 기존 테스트의 정합성을 전수 확보하고, 서버 사양에서 제거되면 런타임 장애로 직결될 deprecated GraphQL 쿼리·필드 10건을 백엔드와 협의해 안전하게 이관했습니다. 모달 시스템은 native dialog 기반 Overlay 프리미티브를 새로 만들되 기존 ModalContainer를 어댑터로 재작성해 29개 소비처를 무중단 이관했고, 파트너웹의 arbitrary 스타일 340여 파일은 「시각 변화 0」을 게이트로 걸고 토큰 구조로 옮겼습니다. 임팩트는 운영 중인 3개 프로덕트가 서비스 중단이나 시각 회귀 없이 새 기반 위로 이동한 것이고, 러닝 포인트는 점진 이관의 핵심이 코드 치환이 아니라 옛 코드와 새 코드가 공존해도 안전한 장치(어댑터·게이트·lint)를 먼저 만드는 일이라는 것입니다.

04
일하는 방식 — AI 에이전트가 읽는 규칙과 가드를 직접 설계해 그 위에서 개발합니다

공고의 「AI-native 개발 문화」를 읽고 가장 먼저 든 생각은, 제가 지금 회사에서 혼자 만들어 쓰고 있는 체계와 같은 언어라는 것이었습니다. 저는 AI 코딩 에이전트를 단순히 쓰는 단계를 넘어, 에이전트가 안전하게 일하도록 규칙과 검증 장치를 먼저 세우는 방식으로 개발합니다.

규칙의 코드화 — 6개 레포를 묶은 워크스페이스 루트에 에이전트가 읽는 규칙 문서(CLAUDE.md·AGENTS.md)를 정본으로 두고, 「단순함 우선 / 임시방편 금지, 근본 원인까지 / 최소한의 영향」 같은 핵심 원칙과 force push 금지·브랜치 전략을 명문화했습니다. 성격이 다른 규칙은 문서를 분리해 폴더·네이밍·아이콘 컨벤션은 docs/CONVENTIONS.md, 라우팅·멀티서비스 구조는 docs/ARCHITECTURE.md로 관리하고, 같은 규칙이 여러 레포에 복사돼 드리프트가 생기지 않도록 「정본은 워크스페이스 한 벌, 하위 레포는 참조만」이라는 cross-repo 문서 동기화 규칙을 세웠습니다.

자동 가드와 검증 — 규칙이 문서로만 남으면 지켜지지 않는다고 보고, 디자인 토큰 위반을 잡는 baseline(ratchet) 방식 lint:tokens를 만들어 CI 게이트로 걸었습니다. 새 위반은 통과할 수 없고 기존 위반만 줄어드는 구조라, 사람이든 에이전트든 규칙을 어기면 머지 전에 걸립니다. 색 대비 AA 검증과 「시각 변화 0」 게이트도 같은 목적의 장치이고, 순수 로직(정렬·병합·페이지네이션, smartcrop focal 좌표, 세션 쿠키 Max-Age)은 단위 테스트와 함께 커밋해 회귀를 잠급니다.

정형화된 흐름과 회고 — 화면 작업은 매번 새로 판단하지 않도록 docs/DESIGN-CHECK.md라는 공통 루틴 한 벌 + 레포별 파라미터 구조로 정리했고, 세션이 끊겨도 다음 작업이 이어지도록 NEXT-SESSION.md와 refactor-log.md로 계획·진행·완료를 추적합니다. 실패에서 배운 것은 docs/lessons/에 회고 문서로 남깁니다 — 「CI 자동화는 영향 범위를 명시해야 한다」, 「위치를 옮긴 문서는 낡은 사실을 품고 온다」 같은 문서가 그 결과물입니다. 덕분에 대량의 레거시 타입 치환처럼 반복적이고 위험한 작업을 AI로 가속하면서도, 교차 검증 단계를 통해 할루시네이션과 잠재 결함 리스크를 통제할 수 있었습니다.

이 체계는 제가 「신중한 설계자」에 가까운 성향이기 때문에 만들어진 것이기도 합니다. 시스템 전체 구조를 먼저 조망하려다 보니 초기 설계에서 속도가 느려질 수 있는데, 규칙과 가드를 미리 세워두면 그 안에서는 마음 놓고 빠르게 갈 수 있다는 것을 배웠습니다. 봉사 상세 화면을 인터랙티브 시안으로 확정한 다음 날 앱에 구현하고 레거시 10개 파일을 함께 걷어낸 것도 그 위에서 가능했던 일입니다.

05
입사 후 포부

파트리지시스템즈에서 함께 일할 기회를 주신다면, 가장 먼저 팀의 규칙 체계와 기존 코드베이스에 형성된 맥락을 정확히 읽는 데 집중하겠습니다. FSD 아키텍처와 팀 컨벤션, 요구사항(REQ)→테스트 케이스(TC)→E2E로 이어지는 검증 루프를 먼저 몸에 익힌 뒤, 제가 검증해 온 운영 역량 — 관측성 정비, 예외 처리 보강, 성능 기준선 수립, 데이터 정합성 점검 — 부터 제 몫의 기여를 시작하겠습니다. 이후에는 사내 디자인 시스템과 공통 컴포넌트 라이브러리 영역에서, 여러 앱이 하나의 언어로 수렴하도록 만들어 본 경험을 그대로 옮겨 팀원분들의 반복 작업을 덜어내겠습니다.

솔직하게 말씀드리면 아직 직접 다뤄보지 않은 영역도 있습니다. Three.js/WebGL·Konva 기반 실시간 렌더링과 WebSocket·Protobuf 스트리밍은 제품 경험이 없습니다. 다만 성능 문제를 다루는 방식 — 체감이 아니라 실측으로 병목을 특정하고, 네트워크 페이로드·렌더 차단 체인·번들 비용을 숫자로 관리하며, 근본 원인까지 내려가는 습관 — 은 렌더링 파이프라인이 무엇이든 동일하게 적용된다고 생각하며, 대용량 데이터를 다루는 화면에서 가장 필요한 감각이라 믿습니다. 빌드 도구 역시 webpack·Turbopack 기반 Next.js에서 번들 분석과 지연 로딩을 다뤄 왔기에 Vite로의 전환은 개념적으로 같은 층위이고, Recoil은 Zustand·Apollo 캐시로 다뤄 온 상태 분리 원칙이, Playwright E2E는 단위 테스트와 CI 게이트를 직접 세워 온 경험이 진입 지점이 되리라 생각합니다. 모르는 것을 아는 척하기보다 빠르게 배워 팀의 속도에 맞추겠습니다.

청소년기를 뉴질랜드에서 보내며 영어로 읽고 쓰고 논의하는 일이 자연스러운 환경에 있었던 만큼, 글로벌 고객사 대응과 멀티테넌트 요구사항 논의에서도 주저 없이 참여하겠습니다. 국내 유일의 피지컬 AI 데이터옵스 플랫폼을 만든다는 목표는 결코 혼자서 이룰 수 없다고 믿습니다. 팀원들과 긴밀히 협업하고 끊임없이 배우며, 파트리지시스템즈가 지향하는 가치를 사용자가 마주하는 화면에서 흔들리지 않는 신뢰성으로 구현해 나가는 가장 든든한 동료가 되겠습니다.
