SELF-INTRODUCTION · 자기소개서
01
지원동기

지난 2년 5개월간 ㈜마이아이비에서 기부·모금·봉사 플랫폼의 프론트엔드를 담당하며, 신뢰가 흔들리는 순간 사용자가 바로 떠난다는 사실을 매일 코드로 마주했습니다. 후원금이 한 번 잘못 결제되면 사용자는 다시 돌아오지 않고, 단체가 한 번 부정확한 정보로 노출되면 플랫폼 전체에 대한 신뢰가 영향을 받습니다. 그래서 단체 상세 페이지 PRD 작성부터 디자인 시스템 토큰화, Sentry 누적 에러 전수 조사까지 화면 한 장의 가설을 정의하는 일부터 그 화면이 안정적으로 돌아가게 만드는 일까지를 한 사람이 이어서 책임지는 방식으로 일해 왔습니다.

토스플레이스를 지원한 이유는 "오프라인 결제 시장에서 디지털 혁신을 만든다"는 회사의 방향이 제가 myib에서 익힌 일하는 방식과 맞닿는 부분이 많기 때문입니다. POS·결제 단말기·키오스크·테이블오더처럼 매장의 주문과 결제가 화면 한 장의 안정성에 직결되는 영역에서는 제품 표면의 안정성과 데이터 정합성이 그대로 사용자 신뢰의 근거가 된다고 생각합니다. 임팩트 도메인에서 쌓은 신뢰 중심의 프론트엔드 아키텍처 설계 및 운영 경험을 바탕으로, 오프라인 결제·매장 운영 영역에서 토스플레이스의 제품 가치를 함께 고도화하고자 지원했습니다.

02
직무 경험 ① — 복잡하고 반복되는 문제를 단순한 형태로 추상화하다

myib 입사 시 코드베이스는 hex 컬러가 수백 곳에 흩어져 있고, native alert() 호출이 58개, 어드민 테이블은 6종이 비표준으로 반복되고 있었습니다. 하나하나 고치는 대신, 반복되는 패턴을 읽어내 단순한 표준 하나로 수렴시키는 접근을 택했습니다. 흩어진 alert 58건은 showToast 단일 패턴으로 일괄 전환하며 그 아래의 useMessage·MessageContext 레이어까지 제거했고, 비표준 테이블 6종은 DataTable·StatusBadge 두 컴포넌트로 추상화해 어드민 테이블 26개 중 22개를 교체했습니다. 화면마다 제각각이던 인라인 빈 상태 13곳과 스피너 7곳도 프리미티브 3종으로 흡수했습니다.

까다로운 사례로는 결제 완료 화면이 있습니다. 정기/일시 결제·매칭·영수증 분기가 뒤엉켜 409줄이 된 화면을, 상태 조합을 다시 정의해 134줄의 단순한 구조로 재설계하면서 헤딩 카피 분기와 "매월 N일 결제 예정" 같은 정보는 오히려 늘렸습니다. 복잡도를 줄이는 것과 요구사항을 줄이는 것은 다르다는 것을 확인한 사례였습니다.

이 추상화들을 개인 습관이 아니라 시스템으로 남기기 위해 banner·button·icon 등 20여 카테고리를 토큰화하고 Storybook 자동 배포 CI와 Figma↔코드 단일 진실 문서(DESIGN.md)를 세웠으며, 최근에는 두 레포의 타이포·radius·shadow·semantic 토큰 집합을 시각 변화 0을 전제로 완전히 일치시키고 baseline 방식 토큰 lint와 CI 게이트로 잠갔습니다. 임팩트는 3개 프로덕트가 같은 언어로 수렴해 신규 화면의 의사결정 비용이 줄어든 것이고, 러닝 포인트는 좋은 추상화는 새 개념을 발명하는 일이 아니라 코드에 이미 반복되고 있는 패턴을 정확히 읽어내는 일에서 시작한다는 것입니다. POS·키오스크·테이블오더·프랜차이즈 대시보드처럼 여러 제품이 하나의 디자인 언어를 공유해야 하는 토스플레이스의 제품군 환경에서도 같은 방식으로 기여할 수 있는 영역이라 생각합니다.

03
직무 경험 ② — 레거시를 새 코드베이스로 점진적으로 이관하다

기존 소스 코드를 새 코드베이스로 옮기는 일을 "한 번의 빅뱅"이 아니라 안전 경로를 설계하는 일로 다뤄 왔습니다. 대표 사례는 Apollo Client v3→v4 대규모 마이그레이션 전담입니다. MockedProvider 사양 변경과 타입 캐스팅 이슈를 추적해 기존 테스트의 정합성을 전수 확보했고, 서버 사양에서 제거되면 런타임 장애로 직결될 deprecated GraphQL 쿼리·필드 10건을 백엔드와 협의해 신규 스키마로 안전하게 이관했습니다. GraphQL 응답을 미러링하던 Zustand store를 걷어내고 useQuery 단일 소스로 재편해 server state와 client state의 경계도 이 과정에서 정리했습니다.

UI 레이어에서도 같은 방식으로 일했습니다. 모달 시스템은 native dialog 기반 Overlay 프리미티브를 새로 만들되 기존 ModalContainer를 어댑터로 재작성해 29개 소비처를 무중단으로 이관했고, 파트너웹의 arbitrary 스타일 340여 파일은 "시각 변화 0"을 게이트로 걸고 토큰 구조로 일괄 이관했습니다. 레거시 admin 디렉토리 190개 파일은 외부 의존성과 함께 단일 커밋으로 격리해 되돌릴 수 있는 형태로 제거했고, 버튼은 FillButton 단일 표준으로 90개 파일을 한 번에 교체했습니다. 임팩트는 이 이관들이 서비스 중단이나 시각 회귀 없이 진행되어 운영 중인 3개 프로덕트가 새 기반 위로 옮겨간 것이고, 러닝 포인트는 점진 이관의 핵심이 코드 치환 자체가 아니라 어댑터·게이트·lint 같은 "옛 코드와 새 코드가 공존해도 안전한 장치"를 먼저 만드는 데 있다는 것입니다.

오프라인 결제 도메인에서는 주문·결제 데이터 정합성과 안정적 운영이 결정적이라고 생각합니다. 결제 view의 주민등록번호·카드 비밀번호 등 민감정보 입력 검증(MIME/확장자 포함)과 SSL 표기 정합성 정리, 안전한 세션 폴링 구조를 확립해 온 경험은, 매장의 주문·결제를 떠받치는 토스플레이스 제품의 운영 신뢰성으로 직접 이어갈 수 있다고 생각합니다.

04
일하는 방식 — 문제를 스스로 발견하고, 고객의 보이스에서 출발한다

주어진 티켓을 처리하는 것보다, 아무도 시키지 않은 문제를 계측으로 드러내는 일에서 성과가 나왔습니다. 프로덕션 홈을 직접 실측해 네트워크 페이로드 18.5MB, LCP 리소스 로드 지연 5,050ms라는 숫자를 만들고 P0~P3 백로그를 스스로 수립했으며, 그 과정에서 앱 전체 SSR 마크업이 비어 있던 근본 원인(세션 로딩 게이트)을 추적해 서버 렌더를 root HTML 0자에서 95KB로 복구했습니다. 의존성 취약점 63건을 10건으로, eslint error 97건을 0건으로 정리한 것도 같은 방식 — 문제를 발견하면 기준선을 세우고 끝까지 내려가는 습관에서 나왔습니다.

문제 정의의 출발점은 고객의 보이스입니다. 단체 상세 리디자인은 "어떤 단체인지 신뢰가 안 간다", "기부 버튼이 어디 있는지 모르겠다"는 CS 응대 기록에서 출발해 문제 세 가지를 정의하고, 진입→검토→결정→결제 퍼널 전체에 North Star와 7개 하위 KPI를 잡은 뒤 그 기획안대로 직접 구현했습니다. 속도 면에서는 봉사 상세 화면을 인터랙티브 시안으로 확정한 다음 날 앱에 구현하고 레거시 10개 파일을 함께 걷어냈으며, 구현이 끝나면 변경 로그와 백엔드 스펙의 갭을 같은 날 기획 문서에 되돌려 기록해 문서가 살아 있는 정합 기준으로 유지되게 합니다.

속도를 위해 Claude 같은 AI 도구를 워크플로우에 내재화하되, 매장의 주문·결제처럼 안정성과 데이터 정합성이 최우선인 도메인 특성을 고려해 AI 산출물은 반드시 교차 검증을 거쳐 할루시네이션과 결함 리스크를 통제합니다. 빠르게 가되, 신뢰를 담보로 잡지 않는 것이 저의 일하는 방식입니다.

05
입사 후 포부

토스플레이스에서 함께 일할 기회를 주신다면, 가장 먼저 팀의 소통 방식과 기존 코드베이스에 형성된 맥락을 정확히 읽는 데 집중하겠습니다. 합류 초기의 가장 빠른 기여는 기존 시스템의 안정적인 운영에 보탬이 되는 것이라 생각하기 때문입니다. POS, 결제 단말기, 키오스크, 테이블오더, 프랜차이즈 대시보드 등 오프라인 매장의 주문·결제를 책임지는 핵심 제품의 도메인 맥락을 빠르게 흡수하고, 오프라인 결제 도메인의 특성상 POS·단말기·프린터·백오피스 사이의 데이터 흐름이 복잡한 만큼 제가 검증해 온 운영 역량 — 관측성 정비, 예외 처리 보강, 데이터 정합성 점검 — 부터 제 몫의 기여를 시작하겠습니다.

이후에는 비즈니스 방향성에 맞추어, 제가 가진 경험으로 팀원분들의 업무 부담을 실질적으로 덜어내겠습니다. 마이아이비에서 제로 베이스부터 단독으로 시스템을 구축하고 요구사항 문서(PRD)를 주도적으로 다듬어본 경험은, 오프라인 결제 시장으로 확장하며 POS·키오스크·테이블오더 제품군을 빠르게 넓혀 가는 단계에서 그대로 옮겨갈 수 있는 자산이 될 것입니다. 또한 평소 연구해 온 AI 협업 워크플로우는 혼자만의 생산성에 가두지 않고, 팀 전체의 개발 부담을 줄이는 공용 자산으로 다듬어 동료들과 함께 성장하는 데 쓰겠습니다.

"오프라인 결제 시장의 디지털 혁신"이라는 토스플레이스의 방향, 그리고 금융의 모든 순간을 새롭게 만든다는 토스커뮤니티의 미션은 결코 혼자서 만들 수 없다고 믿습니다. 팀원들과 긴밀히 협업하고 끊임없이 배우며, 토스플레이스가 지향하는 가치를 사용자가 마주하는 화면에서 흔들리지 않는 신뢰성으로 구현해 나가는 가장 든든한 동료가 되겠습니다.
