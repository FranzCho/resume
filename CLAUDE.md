# CLAUDE.md

이 폴더는 이직용 자소서 작성 시스템이다. 마스터 템플릿·변수표·워크플로우·회사별 데이터가 이 파일 하나에 모여 있다. 새 회사 자소서를 쓸 때 이 파일만 참조하면 된다.

---

## 요청별 분기 (가장 먼저 확인)

사용자 요청 표현에 따라 산출물과 위치가 다르다.

- **"이력서 만들어줘"** → 루트의 공통 이력서 템플릿 `resume.html` 을 기준으로 복사해 새 멀티파일 지원서를 만든다.
  - 저장 위치: `applications/{회사명}_{YYYYMMDD}/지원서_{회사명}_{YYYYMMDD}.html`
  - 템플릿의 레이아웃·경력기술서 부분은 유지하고, **자기소개서(자소서) 부분만 해당 회사 공고에 맞게 수정**한다.
  - 자소서 본문은 아래 "자소서 마스터 본문" + 변수표 워크플로우를 그대로 적용해 작성.
  - ⚠️ **이력서 요청에도 "회사별 채운 값" 표 추가는 필수** — 지원서만 만들고 표를 빠뜨리면 이 파일의 단일 참조 전제가 깨진다. (과거 누락 사례: 아미쿠스렉스 20260609, 앤톡 20260630 → 2026-07-07 백필 완료)
- **"자기소개서만 만들어줘" / "자소서만 만들어줘"** → `cover-letters/자소서_{회사명}_{YYYYMMDD}.md` 로 자소서 단일 파일만 만든다. (HTML 이력서는 만들지 않음)

> 공통 이력서 기준 파일 = `resume.html` (루트의 공통 이력서 템플릿). 이력서 요청 시 항상 이 파일을 베이스로 시작한다.

```
resume/
├── CLAUDE.md                       # 매뉴얼 + 마스터 본문 + 변수 정의 + 회사별 값 (이 파일)
├── resume.html                     # 공통 이력서 템플릿 — "이력서 만들어줘" 시 항상 이걸 복사해서 시작
├── 경력기술서_20260714.txt           # 회사 무관 정본 — 모든 회사에 그대로 재사용
├── 리멤버_경력기술서_20260714.txt      # 리멤버 플랫폼 제출용 변형본 (20260714 정본 기반)
├── 경력기술서_20260617.txt           # 구버전 (보관용 — 신규 사용 금지, 리멤버_20260617 포함)
├── 경력기술서_20260601.txt           # 구버전 (보관용 — 신규 사용 금지)
├── cover-letters/                  # 회사별 자소서 사본 (텍스트 단일 파일)
│   └── 자소서_{회사명}_{YYYYMMDD}.md
└── applications/                   # 멀티파일 지원서 (HTML/사진 등 포함)
    └── {회사명}_{YYYYMMDD}/
```

## 새 회사 자소서 작성 워크플로우

1. **공고에서 정보 수집** — 회사명·직무명·미션/비전 인용·도메인·핵심 제품 2~3개·회사 로드맵
2. **아래 "회사별 채운 값" 섹션에 새 회사 표 추가** — 12개 변수 모두 채움
3. **아래 "자소서 마스터 본문" 을 새 파일로 복사** — `cover-letters/자소서_{회사명}_{YYYYMMDD}.md`
4. **사본에서 `{{변수}}` 모두 치환** — 변수표 값으로 (에디터의 find & replace 또는 수동)
5. **조사 자연스럽게 조정** — 회사명 받침 유무에 따라 "을/를", "은/는", "이/가" 다듬기
6. **회사 특화 시그널 점검** — 공고 우대사항을 본문에 자연스럽게 매핑 가능한지 확인
   - 예: SSR 우대 → [3] 마지막 문장에 *"일관된 SSR 구조"* 강조
   - 예: 성능 최적화 우대 → [2]의 *"빌드 비용 감소"* 부분 살짝 강조

## 파일명 규칙

- 자소서 사본: `자소서_{회사명}_{YYYYMMDD}.md` (예: `자소서_누아_20260608.md`)
- 멀티파일 지원서: `applications/{회사명}_{YYYYMMDD}/` 폴더로 묶고 그 안에 `지원서_{회사명}_{YYYYMMDD}.{md,html,png}` 형태로 저장
- 면접 준비 자료: 같은 `applications/{회사명}_{YYYYMMDD}/` 폴더에 `면접준비_{회사명}.md`, `모의문답_{회사명}.md` 형태로 저장

## 경력 기간 표기 규칙

입사일 = **2024.02**. 모든 문서의 경력 기간("N년 M개월")은 **작성일 기준 만 개월수로 재계산**해서 쓴다 (올림 금지).

- 예: 2026-06-09 작성 → 2년 4개월 / 2026-07-07 작성 → 2년 5개월
- 적용 대상: 자소서 [1] 첫 문장(`{{CAREER_DURATION}}`), resume.html의 경력사항 헤더·career-period 뱃지, 경력기술서 첫 줄
- 배경: 2026-06-04~22 작성 자소서들이 실제 2년 4개월 시점에 "2년 5개월"로 표기된 드리프트가 있었음 — 재발 방지용 규칙

## [2. 직무 경험 ① - 디자인 시스템을 코드에서 끌어올린 1년]

이 원칙으로 약 1년에 걸쳐 banner·button·icon·selector·table·toast 등 20여 개 카테고리를 토큰화하고, Storybook을 구축해 GitHub Pages에 자동 배포되도록 CI를 세팅했습니다. 디자이너·개발자가 매번 새로 합의하지 않아도 되도록 Figma↔코드 매핑을 단일 진실 문서(DESIGN.md)로 도입했습니다. 어드민 영역에서는 DataTable·StatusBadge를 신규 구축한 뒤 두 번째 내부 어드민 시스템(admin2)의 26개 테이블 중 22개를 표준 컴포넌트로 교체했고, 흩어져 있던 native alert 58건은 일관된 showToast 패턴으로 일괄 전환하면서 사용되지 않던 useMessage 및 MessageContext 레이어까지 함께 제거했습니다.

이와 병행해 레거시 admin 디렉토리 190개 파일을 외부 의존성과 함께 단일 commit으로 이관·정리했고, 하루 동안 미사용 컴포넌트·유틸·public 정적 파일 약 320여 개를 일괄 삭제했습니다. 아이콘 영역에서는 카테고리별 100여 개를 약 60개로 통합(40% 절감)하면서 PNG 16종을 SVG 컴포넌트로 전환했고, 버튼은 FillButton 단일 표준으로 통일하며 90개 파일을 한 번에 교체해 코드베이스의 부피와 빌드 비용을 함께 줄였습니다.

이 작업은 화려한 신규 기능은 아니었지만, 결과적으로 B2C 앱·B2B 파트너·어드민 3개 프로덕트가 같은 토큰과 컴포넌트로 수렴하면서 신규 화면 하나를 만드는 데 드는 의사결정 비용이 줄었습니다.

## [3. 직무 경험 ② - 기획서를 직접 쓰고 그 위에서 코드를 짠다]

프론트엔드 개발자로 일해왔지만, 화면 한 장을 만들 때마다 사용자가 어디서 들어와 무엇을 결정하고 어디로 빠져나가는지를 먼저 그려보는 사람입니다. 기능 명세를 받아 구현하기보다, 유저의 의사결정 동선·고객사(단체·기업)의 노출 요구·웹에서의 표현 방식 세 축을 한 문서에서 함께 설계한 뒤 자기 손으로 구현하는 사이클로 일합니다. 의견이 갈리는 상황에서도 화면 한 장이 놓일 시스템 전체에서 무엇이 맞춰져야 하는지를 함께 보고 하나씩 합의를 만들어가는 방식이, 화면이 제 기능을 하도록 만든다고 믿습니다.

기획서의 입력은 고객지원 채널에 누적된 사용자 문의, 진입 경로별 동선 분석, 백엔드·운영·마케팅 담당자와의 직접 협의 세 채널에서 옵니다. 대표 사례인 단체 상세 페이지 리디자인 PRD(v2.0)에서는 "어떤 단체인지 신뢰가 안 간다", "기부 버튼이 어디 있는지 모르겠다" 같은 CS 응대 기록을 출발점으로 '신뢰 신호 분산 / CTA 휘발성 / 임팩트 추상' 세 가지 문제를 정의했고, 화면 단위가 아니라 진입, 검토, 결정, 결제로 이어지는 퍼널(Funnel) 전체에서 North Star와 7가지 하위 KPI(Sub-KPI)를 잡았습니다. 페르소나를 세 그룹으로 세분화하고, 4단계 결제 흐름을 Mermaid 플로우 다이어그램과 7개 화면 인터랙티브 프로토타입으로 옮겨 동료들이 클릭만으로 가설을 검토할 수 있도록 했습니다. 마이페이지 기획서에서는 9개 메뉴 × 5개 상태 케이스(약 45 조합)를, 모금 상세 리디자인에서는 모금 상태·매칭·신뢰도·콘텐츠를 조합한 100여 가지 케이스를 같은 방식으로 정리했습니다.

동일한 책임 범위를 바탕으로 B2B 파트너 프론트엔드 시스템을 Next.js 기반으로 제로 베이스에서 단독으로 셋업하며 크로스 기능 조직 간 도메인 인터페이스 설계 합의를 주도했습니다. Apollo Client v4 마이그레이션 시 MockedProvider 사양 변경과 타입 캐스팅 이슈를 해결하고, 서버 장애로 직결될 수 있는 deprecated GraphQL 쿼리 10건을 신규 스키마로 안전하게 이관했으며, Sentry 누적 에러 23건의 전수 조사와 예외 처리 보완까지 완수해 화면 위의 가설부터 하부 데이터·관측성 레이어의 정합성까지 전 과정을 책임지는 엔지니어링 역량을 증명했습니다. 검색 노출 측면에서도 Next.js App Router의 generateMetadata API를 활용해 단체·모금·봉사·캠페인 등 도메인별 메타데이터와 Open Graph 정보를 일관된 구조로 정리했습니다.
