## 2. 경력 불릿 — 하나씩 풀어서 (20260707 기준 8개 · 20260714/17 추가분은 §7)

### ① B2B 파트너 프론트엔드 0→1 단독 구축

**무슨 일**: 원래 파트너(단체)들이 쓸 전용 웹이 없었다. Next.js로 레포 생성부터 인증·세션 미들웨어, 캠페인 등록·관리, 사업자등록증 기반 신원 인증, 프로필 자동 프리필, 정산 조회·명세 모달·엑셀 다운로드까지 혼자 설계하고 출시했다. 메인 앱의 세션 토큰 갱신 패턴을 파트너 시스템으로 이식했고, 결제 view의 주민등록번호·카드 비밀번호 등 민감정보 입력 검증(MIME/확장자 포함)도 직접 처리했다.

**왜 어필 포인트**: "0→1을 혼자"는 시니어급 신호다. 기술 선택→아키텍처→보안→출시를 전부 스스로 결정했다는 뜻.

**예상 꼬리질문**
- "기술 선택 기준은?" → 메인 앱과의 디자인 시스템·토큰 호환성 확보가 최우선이라 Next.js + Tailwind 결합, 멀티 레포 간 자산 호환 구조로 설계.
- "보안은 어떻게?" → 미들웨어 토큰 갱신 + 미인증 API 호출 가드 + 에러 기반 리다이렉트, refresh token rotation, 민감정보 입력 검증. 안전한 세션 폴링 구조.
- "혼자 하면서 가장 어려웠던 점?" → (내 언어로 준비해 둘 것 — 솔직한 사례 하나)

### ② 핵심 화면을 PRD로 직접 기획하고 그대로 구현

**무슨 일**: 단체 상세 페이지가 대표 사례. CS 응대 기록("어떤 단체인지 신뢰가 안 간다", "기부 버튼이 어디 있는지 모르겠다")에서 출발해 '신뢰 신호 분산 / CTA 휘발성 / 임팩트 추상' 3개 문제로 정의 → 진입·검토·결정·결제 퍼널 전체에 North Star + 7 Sub-KPI 설정 → Mermaid 플로우 + 7화면 클릭 프로토타입으로 동료 검토 → 그 기획안대로 직접 구현. 결제 view를 페이지 이동 없이 인라인화했고 CheckoutSuccess는 409→134줄로 재설계.

**왜 어필 포인트**: "기획 가능한 개발자"는 초기 스타트업·1인 포지션에서 가장 희소한 카드. 지표(KPI)로 말하는 개발자라는 증거.

**예상 꼬리질문**
- "North Star가 정확히 뭐였나?" → 상세 페이지 진입 → 결제 완료 전환. Sub-KPI는 CTA 클릭률·단계별 이탈률·정기기부 비중 등 7개 측정 사양까지 설계.
- "PRD 이후 실제 지표가 개선됐나?" → **정직하게**: 측정 사양까지 설계했고 전환 흐름 정비(결제 인라인화)까지 구현 완료. 개선 폭의 정량 추적은 이후 과제였다 — 부풀리지 말 것.

### ③ 디자인 시스템 + 토큰 거버넌스

**무슨 일**: 입사 시 hex 컬러 수백 곳, alert() 58개, 비표준 테이블 6종. 디자이너에게 새 시스템을 받아오는 top-down 대신, 코드의 실사용처를 카테고리별로 추출하는 **bottom-up**으로 20여 카테고리 토큰화. Storybook을 GitHub Pages에 자동 배포하는 CI 구축, Figma↔코드 매핑을 DESIGN.md 단일 진실 문서로. 파트너웹은 감사(DESIGN-AUDIT)→토큰 신설→primitive 구축→일괄 이관 4단계로 정비.

**왜 어필 포인트**: 1인인데도 "시스템으로 일한다"는 증거. lint:tokens 같은 자동화 거버넌스는 규모 있는 팀에서도 드물다.

**예상 꼬리질문**
- "왜 bottom-up?" → 이미 형성된 패턴을 무시한 새 시스템은 이관 비용만 만든다. 실사용처 추출이 먼저.
- "토큰 위반은 어떻게 막나?" → lint:tokens 커스텀 린트로 CI에서 차단, 0 에러 유지.

### ④ 번들·빌드 비용 축소

**무슨 일**: 레거시 admin 190파일을 의존성 포함 단일 커밋으로 이관(롤백 가능하게), 미사용 320여 개를 의존성 그래프로 확인 후 삭제(회귀 없음), 아이콘 40% 통합, PNG→SVG, 버튼 90파일 표준화.

**주의**: 빌드 시간·번들 크기의 **전후 실측치는 없다**. "빌드 비용을 줄였다"까지만 말하고, 몇 %냐고 물으면 "정돈 규모(파일 수) 기준으로 관리했고 정량 벤치마크는 남기지 못한 게 아쉬운 점"이라고 정직하게.

### ⑤ 상태관리·데이터 레이어·관측성

**무슨 일**: client state(Zustand 14 store)와 server state(Apollo 캐시)를 분리 운영. Apollo v3→v4 마이그레이션을 app-front에서 전담. deprecated GraphQL 쿼리 10건을 서버 제거 전에 선제 이관. 4개 레포 Sentry 표준화 + 에러 23건 전수 조사.

**⚠️ 정확하게 말하기**: v4 마이그레이션은 **app-front**. partner-front는 아직 Apollo 3.13이다. "React Query 써봤나?" → "직접 사용은 없지만 같은 server state 카테고리인 Apollo를 마이그레이션 수준까지 다뤘다"로 (CLAUDE.md 실리콘투·스펙터 답변 가이드 참조).

### ⑥ 마이페이지 대시보드형 재구축

**무슨 일**: 정기기부 화면을 단순 탭형에서 통계 카드·구독 리스트·상태 배너를 갖춘 대시보드형으로 재설계. 기부·포인트·매칭 내역 3종을 실데이터로 통합, 포인트 내역은 563줄 전면 재작성. 재사용 섹션 아키텍처 + 아이콘 11종 정비.

**왜 어필 포인트**: "데이터를 사용자가 직관적으로 이해하는 화면"의 대표 사례 — 대시보드·어드민 성격 회사(앤톡 등)에 직결.

### ⑦ 결제 확인서·정산 출력 일원화

**무슨 일**: 기부 확인서 PDF(미리보기, 로고 워터마크, 신청/완료 자동 구분)와 정산 엑셀 출력을 exceljs 공용 빌더 하나로 통합. 서류 발급은 세무·증빙과 연결되는 기능이라 정확성이 핵심.

### ⑧ 모바일·SEO·운영

**무슨 일**: login·signup·legal + 정본 Modal을 mobile-first로 재작성하되 **로직·props는 무변경** (UI만 안전하게 교체했다는 뜻). 360px gutter 계산, dvh, safe-area-inset-bottom, sticky CTA. generateMetadata로 도메인별 메타데이터·Open Graph를 lib/metadata.ts 한 곳에서 제어. 모금 만들기에 LLM 자동채우기 연동(사용자가 AI 출력을 검토·수정·재요청하는 인터페이스). react-email 메일 8종, CKEditor 5 커스터마이징.

## 4. 정직하게 말해야 하는 것들 (공통 미스매치)

부풀리면 꼬리질문 한 번에 무너진다. 회사별 상세 답변은 CLAUDE.md 각 회사 섹션 참조.

- **테스트 코드 직접 작성 (Jest/Vitest/TDD)**: 직접 작성 경험 없음. Apollo MockedProvider 정합성 확보 + Playwright·chrome-devtools MCP 검증 워크플로우로 우회 답변. "학습 후 진입 가능"까지만.
- **i18n/다국어**: 직접 적용 경험 없음. 메타데이터·SSR 도메인별 분리 경험으로 빠른 학습 가능 프레임.
- **React Query/TanStack Query**: 직접 사용 없음. Apollo(같은 server state 카테고리) 마이그레이션 전담 경험으로 연결.
- **경력 연차 (3년+ 공고)**: 2년 5개월. "1인으로 4개 프로덕트·2,200+ 커밋의 실무 밀도" 프레임.

## 6. 용어 미니 사전 (설명 요구받을 수 있는 것들)

- **North Star Metric**: 제품의 성공을 대표하는 단 하나의 지표. 하위에 Sub-KPI를 두고 분해 측정.
- **디자인 토큰**: 색·간격·타이포 같은 디자인 값을 이름 붙은 변수로 관리하는 것. "arbitrary 값"은 토큰 없이 하드코딩된 값.
- **primitive 컴포넌트**: Button·Modal·Field처럼 도메인 로직 없는 최소 단위 정본 컴포넌트.
- **MockedProvider**: Apollo의 테스트용 mock 프로바이더. v4에서 사양이 바뀌어 기존 테스트 정합성을 전수 확보했다는 맥락.
- **deprecated 쿼리**: 백엔드가 폐기 예정으로 표시한 GraphQL 필드. 서버에서 제거되는 순간 프론트가 런타임 장애 — 그래서 선제 이관이 중요.
- **관측성(Observability)**: 에러·성능을 배포 후에도 볼 수 있는 체계. 여기서는 Sentry.
- **generateMetadata**: Next.js App Router에서 페이지별 메타데이터(SEO·Open Graph)를 서버에서 동적 생성하는 API.
- **refresh token rotation**: 리프레시 토큰을 1회용으로 교체 발급해 탈취 피해를 줄이는 패턴.
- **세션 폴링**: 결제 등 긴 흐름에서 세션 유효성을 주기적으로 확인하는 구조.
